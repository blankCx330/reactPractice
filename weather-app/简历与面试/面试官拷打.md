### 一、React 基础与进阶（难度：⭐⭐）

#### Q1: 你的组件是怎么拆分的？为什么这样拆？

**标准答案期望**：
- 三层结构：WeatherApp（状态管理）→ Container（布局）→ Feature Components（具体功能）
- 职责分离：状态提升到顶层，容器组件负责布局，展示组件只渲染数据

**追问**：
- 如果组件层级再深，props 传递会不会很麻烦？
- 你知道什么方案可以解决这个问题吗？

**参考答案**：
> Props 层层传递确实会造成"prop drilling"问题。可以用 Context API 创建 LocationContext，或者使用 Zustand 这样的轻量状态管理库，直接在需要的组件中获取位置状态。

---

#### Q2: 你用了哪些 React Hooks？能讲讲 useEffect 的依赖数组吗？

**标准答案期望**：
- 用了 `useState`、`useEffect`、`useQuery`（TanStack Query 的 Hook）
- 依赖数组决定 effect 何时执行：空数组 `[]` 只执行一次，有依赖 `[userLocation]` 则依赖变化时执行

**追问**：
- 你代码里有 `useEffect(() => { if (userLocation) { setLon(...) } }, [userLocation])`，为什么要这样做？
- 如果不加 `userLocation` 到依赖数组会怎样？

**参考答案**：
> 因为 `userLocation` 是通过 `useQuery` 异步获取的，初始值是 undefined。当定位成功后，`userLocation` 才有值，需要 `useEffect` 监听变化来更新 `lon` 和 `lat`。如果不加依赖，effect 只在挂载时执行一次，`userLocation` 还是 undefined，无法正确更新。

### 二、TanStack Query 深度考察（难度：⭐⭐⭐）

#### Q4: 为什么选择 TanStack Query？它解决了什么问题？

**标准答案期望**：
- 替代传统的 fetch + useState + useEffect 模式
- 自动处理：loading 状态、error 状态、缓存、重试、去重、轮询
- 减少样板代码，更关注业务逻辑

**追问**：
- 你用到了哪些 TanStack Query 的特性？
- `enabled` 是做什么的？

**参考答案**：
> 我用到了：
> - `enabled`：条件查询，只有经纬度有效时才发起请求
> - `staleTime`：数据新鲜度，5分钟内不会重新请求
> - `retry`：失败后自动重试
> - `queryKey`：缓存键，依赖变化时自动重新请求
>
> `enabled` 让我可以控制查询的执行时机。比如用户还没输入城市名时，我不希望发起无效请求，就设置 `enabled: inputCityName.length > 1`。

---

#### Q5: 你的 `staleTime` 设置了 5 分钟，这个值是怎么确定的？

**标准答案期望**：
- 天气数据更新频率通常在 10-30 分钟
- 5 分钟是平衡新鲜度和性能的选择

**追问**：
- 如果用户想要手动刷新呢？
- `staleTime` 和 `cacheTime`（现在叫 `gcTime`）有什么区别？

**参考答案**：
> 可以通过 `refetch()` 方法手动刷新，或者给刷新按钮绑定 `refetch`。
>
> `staleTime` 是数据"新鲜"的时间，新鲜的数据不会重新请求；`gcTime` 是数据在缓存中保留的时间，超过这个时间会被垃圾回收。默认 `staleTime: 0`，`gcTime: 5min`。

---

#### Q6: 如果用户快速切换城市，多个请求同时发出，会有什么问题？

**标准答案期望**：
- 竞态问题：后发出的请求可能先返回，显示错误的城市天气
- TanStack Query 会自动处理这个问题

**追问**：
- 它是怎么处理的？
- 如果用原生 fetch 怎么解决？

**参考答案**：
> TanStack Query 会自动取消过时的请求，保证 `queryKey` 对应的最新请求生效。
>
> 如果用原生 fetch，需要用 AbortController：
> ```tsx
> useEffect(() => {
>   const controller = new AbortController()
>   fetch(url, { signal: controller.signal })
>   return () => controller.abort()  // 清理时取消请求
> }, [city])
> ```

### 三、TypeScript 考察（难度：⭐⭐⭐）

#### Q7: 你的类型定义文件有 460 多行，是怎么组织的？

**标准答案期望**：
- 基础类型 `QWeatherBaseResponse` 作为基类
- 各 API 响应类型继承基础类型
- 嵌套的数据结构单独定义（如 `WeatherNow`、`Location`）

**追问**：
- 为什么要用 `extends` 继承？
- 你代码里有 `any` 类型，为什么？应该怎么改？

**参考答案**：
> 继承可以复用公共字段（`code`、`refer`），避免重复定义，也更清晰地表达"这是一个和风天气响应"的语义。
>
> `any` 是偷懒的做法，应该定义具体类型：
> ```tsx
> // 错误
> userLocation?: any
> 
> // 正确
> import type { UserLocation } from './types/qweather'
> userLocation?: UserLocation
> ```

---

#### Q8: 你的 `WeatherNow` 接口里有些字段是可选的（`cloud?`、`dew?`），为什么？

**标准答案期望**：
- 这些字段在 API 文档中标注"可能为空"
- 使用 `?` 表示可选属性，TypeScript 会要求在使用前检查

**追问**：
- 如果我直接访问 `data.now.cloud`，TypeScript 会报错吗？
- 应该怎么安全地使用？

**参考答案**：
> 不会报错，因为 `cloud` 是可选属性，类型是 `string | undefined`。但直接使用可能得到 undefined。
>
> 安全使用方式：
> ```tsx
> // 可选链
> const cloud = data.now.cloud?.toString()
> 
> // 默认值
> const cloud = data.now.cloud ?? '未知'
> ```

### 四、API 与网络请求（难度：⭐⭐）

#### Q9: 你的 API 调用分散在多个组件里，这样有什么问题？

**标准答案期望**：
- 难以维护：API 地址、参数格式分散各处
- 难以复用：同一 API 在多处调用时重复代码
- 难以统一处理：错误处理、请求拦截无法统一

**追问**：
- 你会怎么改进？
- 你知道 axios 的拦截器吗？

**参考答案**：
> 应该抽离为独立的 service 层：
> ```tsx
> // api/request.ts
> export async function fetchAPI<T>(endpoint: string, params: Record<string, string>) {
>   const url = new URL(`https://${API_HOST}${endpoint}`)
>   Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v))
>   url.searchParams.set('key', API_KEY)
>   
>   const res = await fetch(url.toString())
>   if (!res.ok) throw new Error(`API Error: ${res.status}`)
>   return res.json() as Promise<T>
> }
> 
> // api/weather.ts
> export const weatherAPI = {
>   getNow: (lon: number, lat: number) => 
>     fetchAPI<WeatherNowResponse>('/v7/weather/now', { location: `${lon},${lat}` })
> }
> ```
>
> axios 拦截器可以在请求前统一添加 token，在响应后统一处理错误。

#### Q10: 你的定位功能如果用户拒绝授权会怎样？

**标准答案期望**：
- 会走到 `reject` 分支，抛出错误
- 当前代码没有处理错误情况

**追问**：
- 应该怎么优雅处理？
- 如何给用户友好的提示？

**参考答案**：
> 应该捕获错误并给用户提示：
> ```tsx
> const useUserLocation = () => {
>   return useQuery<UserLocation>({
>     queryKey: ['userLocation'],
>     queryFn: async () => {
>       try {
>         const position = await getLocation()
>         return { lat: position.coords.latitude, lon: position.coords.longitude }
>       } catch (error) {
>         // 用户拒绝授权或浏览器不支持
>         console.error('定位失败', error)
>         throw new Error('无法获取位置，请手动选择城市')
>       }
>     },
>     retry: false,  // 定位失败不重试
>   })
> }
> ```
>
> 在 UI 上显示提示信息，并引导用户手动选择城市。

**追问**：

> 优先改进：
> 1. 添加 Loading 状态和错误提示（用户体验最重要）
> 2. 清理 `console.log` 和修复 `any` 类型（代码质量）
> 3. 添加 Error Boundary（错误边界）
>
> Error Boundary 是 React 的错误处理机制，可以捕获子组件的渲染错误，显示降级 UI：
> ```tsx
> class ErrorBoundary extends Component {
>   state = { hasError: false }
>   static getDerivedStateFromError() { return { hasError: true } }
>   render() {
>     return this.state.hasError ? <div>出错了，请刷新</div> : this.props.children
>   }
> }
> ```

### 六、算法与编程（难度：⭐⭐⭐）

#### Q14: 你的城市搜索是输入 2 个字符才触发，为什么？

**标准答案期望**：
- 避免输入 1 个字符就请求，减少无效请求
- `enabled: inputCityName.length > 1`

**追问**：
- 用户输入过程中会频繁触发请求，有没有更好的方案？
- 你知道防抖（debounce）吗？

**参考答案**：
> 可以用防抖优化，用户停止输入 300ms 后才发起请求：
> ```tsx
> import { useDebouncedValue } from './hooks/useDebouncedValue'
> 
> const [input, setInput] = useState('')
> const [debouncedInput] = useDebouncedValue(input, 300)
> 
> useQuery({
>   queryKey: ['city', debouncedInput],
>   enabled: debouncedInput.length > 1
> })
> ```

#### Q15: 你的 24 小时预报数据需要渲染 24 个卡片，如果变成 168 小时（7天），会有性能问题吗？

**标准答案期望**：
- 24 个还好，但 168 个可能会有性能问题
- 需要考虑虚拟列表

**追问**：
- 什么是虚拟列表？原理是什么？
> 虚拟列表只渲染可视区域内的元素，通过计算滚动位置动态更新渲染内容。比如 168 条数据，可视区域只能显示 10 条，就只渲染 10 条，大大减少 DOM 节点。
>
> 可以用 `react-window` 库实现。

## 📝 面试官视角总结

### 项目亮点
1. **技术栈现代化**：React 19 + TypeScript + Vite，跟上主流
2. **TanStack Query**：不是简单的 fetch，体现了对数据管理的思考
3. **TypeScript 类型定义**：460 行类型，有类型意识
4. **错题本**：少见的学习习惯，值得肯定

### 项目不足
1. **工程化欠缺**：没有测试、没有错误边界、没有 CI/CD
2. **代码质量**：`console.log` 未清理、`any` 类型、缺少 `key`
3. **用户体验**：没有 Loading 状态、没有错误提示
4. **架构设计**：API 调用分散、没有自定义 Hooks

### 面试建议
- **扬长避短**：多讲 TanStack Query 的配置和作用，少讲简单功能
- **诚实谦虚**：主动说出项目不足，展示反思能力
- **准备改进方案**：被问到不足时，能说出怎么改
- **展示错题本**：这是亮点，可以主动提
