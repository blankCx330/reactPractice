### 问题 #5: 经纬度默认值搞反导致API请求失败

**首次记录**: 2026-03-12 00:00
**最近记录**: 2026-03-12 00:00
**记录次数**: 1

**原始问题**:

> 为什么会出现这个错误？(API返回400 Bad Request)

**相关代码**:

```tsx
const [lon, setLon] = useState(userLocation?.lon ?? 39) // 错误！39是纬度
const [lat, setLat] = useState(userLocation?.lat ?? 116) // 错误！116是经度
```

**问题分析**:
经纬度默认值搞反了。北京坐标是经度116、纬度39，但代码把39赋给了lon，116赋给了lat。导致API请求变成`location=39,116`（错误的格式），服务器返回400错误。

**正确做法**:

```tsx
const [lon, setLon] = useState(userLocation?.lon ?? 116) // 经度
const [lat, setLat] = useState(userLocation?.lat ?? 39) // 纬度
```

**学习要点**: ⭐ 经度范围-180~180，纬度范围-90~90。API格式`location=经度,纬度`。北京是经度116、纬度39

```tsx

```

**问题分析**:
React要求列表中的每个元素都有唯一的key属性，用于高效更新DOM。缺少key会导致警告，且可能引发渲染问题。

**学习要点**: ⭐ `.map()`返回的元素必须添加`key`属性，通常用数据的唯一标识（如id、日期等）

### 问题 #7: 在 queryFn 内部使用 isLoading/isError

**首次记录**: 2026-03-14 15:00
**最近记录**: 2026-03-14 16:00
**记录次数**: 2

**原始问题**:

> 为什么不能在 queryFn 里写 if(isLoading)？

**历史问题**:

> - [2026-03-14 16:00] 那为什么这里又有可以判断状态？queryFn 里写 if(isError) 和 if(isLoading)

**相关代码**:

```tsx
const { isLoading, data } = useQuery({
  queryFn: async () => {
    const position = await getLocation()
    if (isLoading) {
      // ❌ 错误！isLoading 还没定义
      return { lat: position.coords.latitude, lon: position.coords.longitude }
    }
  },
})
```

**问题分析**:
`isLoading`/`isError` 是 `useQuery` **返回**的状态，在 `queryFn` 执行时还不存在。执行顺序是：调用 useQuery → 执行 queryFn → 返回 { isLoading, ... }。在 queryFn 里访问这些状态相当于在变量定义之前使用它。

**重要**：用户看到的"可以判断状态"的代码**同样是错误的**，不会正常工作：

```tsx
queryFn: async () => {
  if (isError) {
    // ❌ 同样错误！isError 未定义
    return <ErrorMessage /> // ❌ queryFn 不应返回 JSX
  }
}
```

这段代码会报错，而且 queryFn 不应该返回组件。

**正确做法**:

```tsx
// queryFn 只负责获取数据，不需要判断状态
const useUserLocation = () => {
  return useQuery<UserLocation>({
    queryKey: ['userLocation'],
    queryFn: async () => {
      const position = await getLocation()
      return {
        lat: position.coords.latitude,
        lon: position.coords.longitude,
      }
    },
  })
}

// 在组件中使用返回的状态
const { data, isLoading } = useUserLocation()
if (isLoading) {
  return <LoadingSpinner />
}
```

**学习要点**: ⭐ `queryFn` 是"干活的人"，`isLoading` 是"干完活后的报告"。干活时报告还没写出来，所以不能在 queryFn 里用 isLoading

### 问题 #4: URL查询参数格式错误

**首次记录**: 2026-03-11 10:30
**最近记录**: 2026-03-11 10:30
**记录次数**: 1

**原始问题**:

> 为什么这段代码无法返回数据？

**相关代码**:

```tsx
const { data: sevenWeather } = useQuery({
  queryKey: ['sevenWeather', lon, lat],
  queryFn: async () => {
    const url = `https://${apiHost}/v7/weather/7d?location=${lon},${lat}/key=${apiKey}`
    return fetch(url).then(res => res.json())
  },
})
```

**问题分析**:
URL 中用 `/key=` 连接查询参数，但 `/` 是路径分隔符，不是查询参数连接符。服务器会把 `/key=xxx` 当成路径的一部分，导致 API 返回错误。

**正确做法**:

```tsx
// 修复：把 /key= 改成 &key=
const url = `https://${apiHost}/v7/weather/7d?location=${lon},${lat}&key=${apiKey}`
```

**学习要点**: ⭐ URL查询参数格式：`?参数1=值1&参数2=值2`。第一个参数用 `?`，后续参数用 `&`，永远不要用 `/`

### 问题 #3: 在查询前修改查询参数的逻辑错误

**首次记录**: 2026-03-10 17:30
**最近记录**: 2026-03-10 18:00
**记录次数**: 2

**原始问题**:

> [2026-03-10 17:30] 为什么我的代码这么修改了还是无法正确的搜索？
> [2026-03-10 18:00] 为什么搜索两次才能输出正确结果？

**相关代码**:

```tsx
const handleKeyDown = (event: React.KeyboardEvent<HTMLInputElement>) => {
  if (event.key === 'Enter') {
    setCityName(getCityLocation?.poi?.[0]?.adm2 ?? '') // 先修改参数
    refetchCityLocation() // 再查询
  }
}
```

**问题分析**:
在触发查询**之前**就修改了查询参数（cityName），导致查询使用的是错误（空）的值。这是典型的"执行顺序错误"：应该是"用当前值查询 → 处理结果"，而不是"修改值 → 用错误值查询"。

**正确做法**:

```tsx
const handleKeyDown = async (event: React.KeyboardEvent<HTMLInputElement>) => {
  if (event.key === 'Enter' && cityName.trim()) {
    // 用当前用户输入的 cityName 查询，不要修改它
    const result = await refetchCityLocation()

    if (result.data?.poi?.[0]) {
      const poi = result.data.poi[0]
      onLocationChange(poi.lon, poi.lat)
    }
  }
}
```

**学习要点**: ⭐ 不要在查询前修改查询参数，也不要在 `setState` 后立即使用该状态。`setState` 是异步的，立即使用会得到旧值。解决方案：直接用变量本身，不要绕状态

### 问题 #9: Zustand 类型定义与 set 用法错误

**首次记录**: 2026-03-15 01:00
**最近记录**: 2026-03-15 01:00
**记录次数**: 1

**原始问题**:

> 这个设置有什么问题？有没有什么更优雅的方法？

**相关代码**:

```tsx
type Store = {
  location: { Location: Location } // ❌ 类型定义错误
  setStore: () => void // ❌ 方法类型不匹配
}

export const LocationStore = create(set => ({
  location: { lon: 116, lat: 39 },
  setStore: (newLocation: Location) => set({ newLocation }), // ❌ set 更新错误
}))
```

**问题分析**:

1. `{Location:Location}` 定义的是"包含 Location 属性的对象"，而不是 Location 类型本身
2. 类型定义无参数，实现却有参数，类型不匹配
3. `set({newLocation})` 是属性简写，等于添加新字段而不是更新现有字段

**正确做法**:

```tsx
type Location = { lon: number; lat: number }

type Store = {
  location: Location // 直接用 Location 类型
  setLocation: (location: Location) => void // 类型要匹配
}

export const LocationStore = create<Store>(set => ({
  location: { lon: 116, lat: 39 },
  setLocation: newLocation => set({ location: newLocation }),
  //                              ↑ 指定要更新的字段名
}))
```

**更优雅的写法**（分开设置经纬度）:

```tsx
type Store = {
  location: Location
  setLocation: (location: Location) => void
  setLon: (lon: number) => void
  setLat: (lat: number) => void
}

export const LocationStore = create<Store>(set => ({
  location: { lon: 116, lat: 39 },
  setLocation: location => set({ location }),
  setLon: lon =>
    set(state => ({
      location: { ...state.location, lon }, // 部分更新
    })),
  setLat: lat =>
    set(state => ({
      location: { ...state.location, lat },
    })),
}))
```

**学习要点**: ⭐ 类型直接用 `Location` 不是 `{Location:Location}`；`set({ field: value })` 要指定字段名；部分更新用回调 `set(state => ({ ... }))`

### 问题 #11: 函数声明语法错误

**首次记录**: 2026-03-15 05:00
**最近记录**: 2026-03-15 05:00
**记录次数**: 1

**原始问题**:

> 缺少返回类型批注的"useCityWeatherData"隐式具有"any"返回类型

**相关代码**:

```tsx
export function useCityWeatherData(lon:number, lat:number) => {
//                                                    ↑↑↑ 多余的 =>
  return useQuery({...})
}
```

**问题分析**:
混用了 `function` 声明和箭头函数语法。`function` 声明不需要 `=>`，`=>` 是箭头函数的语法。

**正确做法**:

```tsx
// 方式一：function 声明（去掉 =>）
export function useCityWeatherData(lon: number, lat: number) {

export const useCityWeatherData = (lon: number, lat: number) => {

}
```

**语法对比**:

| 声明方式      | 语法                    | 需要 =>？ |
| ------------- | ----------------------- | --------- |
| function 声明 | `function name() {}`    | ❌ 不需要 |
| 箭头函数      | `const name = () => {}` | ✅ 需要   |

**学习要点**: ⭐ `function name() {}` 没有 `=>`，`const name = () => {}` 才有 `=>`

### 问题 #12: 在事件处理函数中调用 Hook

**首次记录**: 2026-03-15 06:10
**最近记录**: 2026-03-15 06:10
**记录次数**: 1

**原始问题**:

> 这两段代码会冲突吗？如果 input 进行输入，getCityLocation 会更新吗？

**相关代码**:

```tsx
const { data: getCityLocation } = useCityLocation(inputCityName)

const handleInputChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setInputCityName(e.target.value)
  useCityLocation(e.target.value) // ❌ 错误！
  setShowSuggestions(true)
}
```

**问题分析**:
在事件处理函数中调用 Hook 是**严重错误**。Hook 必须在组件顶层调用，不能在事件处理函数、循环、条件语句中调用。React Query 会自动响应状态变化，不需要手动调用。

**正确做法**:

```tsx
// 组件顶层调用 Hook
const { data: getCityLocation } = useCityLocation(inputCityName)

const handleInputChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setInputCityName(e.target.value) // 只更新状态
  setShowSuggestions(true)
  // 不需要手动调用 useCityLocation！
}

// 数据流：
// setInputCityName() → 状态更新 → 组件重渲染
// → useCityLocation(新值) → 自动重新查询 → getCityLocation 更新
```

**Hook 调用规则**:

| ❌ 错误位置  | ✅ 正确位置    |
| ------------ | -------------- |
| 事件处理函数 | 组件顶层       |
| 循环体内     | 函数体开头     |
| 条件语句内   | 所有 Hook 之前 |

**学习要点**: ⭐ Hook 只能在组件顶层调用；状态变化会自动触发 Hook 重新执行；不需要在事件处理函数中手动调用 Hook

### 问题 #13: queryKey 未包含动态参数导致查询不更新

**首次记录**: 2026-03-15 06:25
**最近记录**: 2026-03-15 06:25
**记录次数**: 1

**原始问题**:

> 删除了 handleInputChange 中的 useCityLocation，但选择列表不会在输入时正确更新

**相关代码**:

```tsx
// useCityLocation.tsx
export const useCityLocation = (cityName: string) => {
  const data = useQuery({
    queryKey: ['cityLocation'], // ❌ 没有包含 cityName！
    queryFn: async () => {
      const cityUrl = `...location=${cityName}...`
      return fetch(cityUrl).then(res => res.json())
    },
    enabled: cityName.length > 1,
  })
  return data
}
```

**问题分析**:
`queryKey` 是固定的 `['cityLocation']`，没有包含动态参数 `cityName`。当 `cityName` 变化时，React Query 认为是同一个查询，不会重新请求，直接返回缓存。

**正确做法**:

```tsx
export const useCityLocation = (cityName: string) => {
  const data = useQuery({
    queryKey: ['cityLocation', cityName], // ✅ 包含动态参数
    queryFn: async () => {
      const cityUrl = `...location=${cityName}...`
      return fetch(cityUrl).then(res => res.json())
    },
    enabled: cityName.length > 1,
  })
  return data
}
```

**修复后的数据流**:

```
输入"北京" → queryKey: ['cityLocation', '北京'] → 请求 API
输入"上海" → queryKey: ['cityLocation', '上海'] → queryKey 变化 → 重新请求 ✅
```

**学习要点**: ⭐ queryKey 必须包含所有动态参数；参数变化时 queryKey 也应该变化；否则 React Query 会认为是同一个查询，不重新请求

### 问题 #14: 搜索建议列表缺少空状态提示

**首次记录**: 2026-03-15 07:00
**最近记录**: 2026-03-15 07:00
**记录次数**: 1

**原始问题**:

> 发现当 getCityLocation 搜索不到城市的时候并没有给用户提示，这非常不好

**相关代码**:

```tsx
{
  showSuggestions && getCityLocation && (
    <div className="suggestions-list">
      {getCityLocation?.location?.map((data: Location) => (
        <div key={data.id} className="suggestion-item">
          {data.adm2} - {data.name} - {data.adm1}
        </div>
      ))}
    </div>
  )
}
```

**问题分析**:
条件渲染只处理了"有结果"的情况，忽略了"无结果"时的用户反馈。当 `location=[]` 时，`map` 返回空数组，div 内没有任何内容，用户不知道是"搜索不到"还是"出了问题"。

**不修复的后果**:
用户输入不存在的城市名，下拉框空空如也，用户困惑：是搜索不到？网络错误？还是正在搜索？可能反复尝试输入，体验极差。

**正确做法**:

```tsx
{showSuggestions && (
  <div className="suggestions-list">
    {/* 情况1：正在加载 */}
    {isFetching && (
      <div className="suggestion-item">搜索中...</div>
    )}

    {/* 情况2：搜索无结果 */}
    {!isFetching && getCityLocation?.location?.length === 0 && (
      <div className="suggestion-item">未找到匹配的城市</div>
    )}

    {/* 情况3：有结果 */}
    {!isFetching && getCityLocation?.location?.map((data: Location) => (
      <div key={data.id} className="suggestion-item" onClick={...}>
        {data.adm2} - {data.name} - {data.adm1}
      </div>
    ))}
  </div>
)}
```

**更优雅的方案（提取渲染函数）**:

```tsx
const renderSuggestions = () => {
  if (!showSuggestions) return null

  if (isFetching) return <div className="suggestion-item">搜索中...</div>

  if (!getCityLocation?.location?.length) {
    return <div className="suggestion-item">未找到匹配的城市</div>
  }

  return getCityLocation.location.map((data: Location) => (
    <div key={data.id} className="suggestion-item" onClick={...}>
      {data.adm2} - {data.name} - {data.adm1}
    </div>
  ))
}

// JSX 中使用
<div className="suggestions-list">{renderSuggestions()}</div>
```

**学习要点**: ⭐ 条件渲染三状态：加载中 → 无结果 → 有结果，缺一不可。用户体验核心是"让用户知道发生了什么"

### 问题 #15: 条件判断逻辑取反错误

**首次记录**: 2026-03-15 07:30
**最近记录**: 2026-03-15 07:45
**记录次数**: 2

**原始问题**:

> 为什么我在输入第二个字的时候页面直接全部清空了？

**历史问题**:

> - [2026-03-15 07:45] 我说的清空是整个页面都不渲染了，不单单指这个列表不渲染，为什么少一个!号会导致这么严重的后果？

**相关代码**:

```tsx
const suggestionsListText = () => {
  if (!showSuggestions) return null

  if (cityLocationIsLoading) return <div>搜索中...</div>

  if (getCityLocation?.location?.length) {  // ❌ 逻辑取反错误！
    return <div>未找到匹配城市</div>
  }

  return getCityLocation?.location.map(data => ...)
}
```

**问题分析**:

条件判断逻辑完全反了！`getCityLocation?.location?.length` 返回数组长度（数字），当长度 > 0 时为 truthy。所以：

- **有数据**（length=2）→ 条件为 true → 显示"未找到匹配城市" ❌
- **无数据**（length=0）→ 条件为 false → 继续执行 map ✅

**为什么会整个页面崩溃？**

缺少 `!` 不仅导致逻辑错误，还会引发 JavaScript 运行时错误：

```
API 返回 { location: null }（某些 API 无结果时返回 null）
    ↓
getCityLocation?.location?.length → null?.length → undefined
    ↓
undefined 是 falsy，不进入 if 块
    ↓
执行 getCityLocation?.location.map(...)
    ↓
location 是 null，调用 null.map() ❌
    ↓
TypeError: Cannot read property 'map' of null
    ↓
React 组件崩溃，整个页面空白！
```

**关键原因**：React 组件中任何 JavaScript 错误都会导致整个组件树崩溃。React 无法确定组件状态是否一致，只能放弃渲染。

**错误传播链**：

```
一个 ! 缺失 → 条件判断逻辑反转 → 错误的数据状态进入下一行 → null.map() 抛出 TypeError → React 捕获错误 → 整个页面空白
```

**正确做法**:

```tsx
const suggestionsListText = () => {
  if (!showSuggestions) return null

  if (cityLocationIsLoading) return <div>搜索中...</div>

  // ✅ 用 ! 取反：无数据时才显示"未找到"
  if (!getCityLocation?.location?.length) {
    return <div>未找到匹配城市</div>
  }

  // 有数据时正常显示列表
  return getCityLocation.location.map(data => (
```

**知识点**: React / 状态管理

**核心解答**:
React 的状态更新是**异步批处理**的。连续调用多次 `setState` 时，React 会把它们合并成一次渲染，中间状态会丢失。

```tsx
setSearchStatus(false) // 请求2
// React 实际执行：直接变成 false，跳过了 true
```

这就像你要从1楼到3楼再到2楼，但电梯只在最终目的地停下——你不会在3楼停留。

**正确做法**:

```tsx
// ❌ 错误：中间状态会丢失
setStatus(true)
setStatus(false)

// ✅ 方式1：用 useEffect 监听状态变化
useEffect(() => {
  if (status) {
    // 执行操作
    setStatus(false) // 完成后重置
  }
}, [status])

// ✅ 方式2：用异步操作
await doSomething()
setStatus(false)

// ✅ 方式3：换一种不依赖状态切换的方案
```

**学习要点**: ⭐ 连续的 `setState` 会被批处理合并，中间状态不会触发渲染。需要用 `useEffect` 监听或 `await` 等待

```tsx
str.padStart(6, ' ') // '   abc'

str.padEnd(2, ' ') // 'abc' (不变)
```

```
0ms    isLoading=true   → 显示 LoadingSpinner（转圈）
```

```tsx
// 为什么这样写？
// 因为 getCurrentPosition 是回调风格，但 React Query 需要 Promise
// 所以用 Promise 包装回调 API，让它能被 await 使用

// 标准模板
const myAsyncFunction = (): Promise<结果类型> => {
  return new Promise((resolve, reject) => {
    // 异步操作
    if (成功) {
      resolve(结果)    // 传递成功值
    } else {
      reject(错误)     // 传递失败原因
    }
  })
}
```

**状态检测**:

```tsx
// 使用 React Query 检测
const { data, isLoading, isError, error } = useUserLocation()

if (isLoading)
  if (isError)
    if (data)
      // Promise 执行中
      // Promise 被 reject
      // Promise 被 resolve

      // 手动检测
      try {
        const position = await getLocation() // resolve 的值
      } catch (error) {
        // reject 的错误
      }
```

**学习要点**: ⭐ `resolve(值)`=成功，`reject(错误)`=失败；必须调用其一；这是包装回调 API 为 Promise 的标准模式；React Query 的 `isError`/`data` 检测 Promise 状态

代码存在**状态冗余**：`userLocation` 已包含位置，又创建了 `lon`/`lat` 状态。`useState` 初始值只执行一次，需要 `useEffect` 同步，容易出错。

**深入理解**:

```tsx
// ❌ 问题：状态冗余，需要手动同步
const [lon, setLon] = useState(userLocation?.lon ?? 116)
useEffect(() => {
  if (userLocation) setLon(userLocation.lon)
}, [userLocation])

// ✅ 方案1：派生状态（不需要手动修改时）
const lon = userLocation?.lon ?? 116
const lat = userLocation?.lat ?? 39

// ✅ 方案2：单一状态源（需要手动修改时）
const [location, setLocation] = useState({ lon: 116, lat: 39 })
useEffect(() => {
  if (userLocation) setLocation({ lon: userLocation.lon, lat: userLocation.lat })
}, [userLocation])
```

**useState 初始值陷阱**:

```tsx

// 后续渲染：userLocation 有值了，但 useState 不会重新执行初始值
const [lon, setLon] = useState(userLocation?.lon ?? 116) // 永远是 116
```

**学习要点**: ⭐ 避免"状态冗余"；能派生就别存状态；`useState` 初始值只执行一次

### 问题 #14: ES6 对象属性简写

**首次记录**: 2026-03-14 19:00

**核心解答**:

**深入理解**:

```tsx
// 两种写法完全等价
const lon = 116

// 结果都是 { lon: 116, lat: 39 }

// 为什么报错？
setLocation({ lon, lat }) // ❌ 找不到变量 lon/lat
```

|------|------|------|
| `{ name }` | 有变量 `name` | `{ name: name的值 }` |

| `{ name: xxx }` | 不需要条件 | `{ name: xxx的值 }` |

**学习要点**: ⭐ 简写条件：属性名 = 变量名 → `{ name }` = `{ name: name }`。前提是必须先定义同名变量

### 问题 #17: 事件处理函数命名规范

```tsx
interface Props {
```

| 切换状态 | `toggle + 名词` | `toggleModal`, `toggleMenu` |

| 特性 | useState | Zustand |
|------|----------|---------|
| 更新方式 | 直接替换 | 合并更新 |

```tsx
// 在组件中使用
function MyComponent() {
```

### 问题 #21: Zustand 缺少泛型导致类型为 unknown

**首次记录**: 2026-03-15 01:30
**最近记录**: 2026-03-15 01:30

```tsx
// ❌ 缺少泛型，state 类型为 unknown

export const LocationStore = create<Store>(set => ({

```

```
create()         → 就像给 TypeScript 一个"盲盒"，它不知道里面是什么
create<Store>()  → 就像给 TypeScript 一张"说明书"，它知道里面有什么
```

**学习要点**: ⭐ Zustand 必须传递泛型 `create<Store>()`，否则 TypeScript 推断为 `unknown`；泛型位置在 `create` 后面的尖括号里

### 问题 #23: useEffect 同步状态的必要性问题

**首次记录**: 2026-03-15 02:30
**最近记录**: 2026-03-15 02:30
**记录次数**: 1

**原始问题**:

> 用 useEffect 监听 userLocation 并更新 location，有必要吗？

**相关代码**:

```tsx
const [location, setLocation] = useState({ lon: 116, lat: 39 })
const { data: userLocation } = useUserLocation()

useEffect(() => {
  if (userLocation) {
    setLocation({ lon: userLocation.lon, lat: userLocation.lat })
  }
}, [userLocation])
```

**知识点**: React / useEffect / 状态设计

**核心解答**:
**不推荐**。这个 useEffect 会自动覆盖用户的主动选择，造成糟糕的用户体验。

**问题演示**:

```
1. 用户打开页面 → location = 北京(默认)
2. 定位成功 → userLocation = 上海 → location = 上海
3. 用户搜索"广州" → location = 广州
4. 定位自动刷新 → userLocation = 上海 → location = 上海 ❌ 覆盖了用户选择！
```

**正确做法**:

```tsx
// 方案1：删除 useEffect，让用户主动点击"定位"按钮时更新
// 用户已经有 handleUserCurrentLocation 按钮处理这个逻辑

// 方案2：只在首次定位成功时初始化
const [hasInitialized, setHasInitialized] = useState(false)

useEffect(() => {
  if (userLocation && !hasInitialized) {
    setLocation({ lon: userLocation.lon, lat: userLocation.lat })
    setHasInitialized(true)
  }
}, [userLocation, hasInitialized])

// 方案3：分离状态
// userLocation → 只读，显示"当前定位"
// location → 用户可控，显示"当前查询位置"
```

**设计原则**:

|------|------|

**学习要点**: ⭐ 不要让自动数据覆盖用户的主动选择；useEffect 同步状态往往是设计问题的信号；让用户主动触发更新而非自动覆盖

### 问题 #25: TypeScript 自定义错误类

**首次记录**: 2026-03-15 03:30
**最近记录**: 2026-03-15 03:30
**记录次数**: 1

**原始问题**:

```tsx
class GeolocationError extends Error {

  }
```

**核心解答**:
这是一个**自定义错误类**，继承自 JavaScript 内置的 `Error` 类，添加额外的 `code` 属性用于错误分类。

**深入理解**:

```tsx

throw new GeolocationError('用户拒绝定位', 1)

//                          ↑ message    ↑ code

// 在 catch 中可以精确判断
try {
  await getLocation()
} catch (error) {
  if (error instanceof GeolocationError) {
    if (error.code === 1) {
      // 用户拒绝 - 提示用户开启权限
    } else if (error.code === 3) {
      // 超时 - 提示重试
    }
  }
}
```

**逐行解析**:

| 行  | 代码                                   | 作用                                |
| --- | -------------------------------------- | ----------------------------------- |
| 1   | `class GeolocationError extends Error` | 声明类并继承 `Error`                |
| 2   | `code: number`                         | 声明属性类型（TypeScript 要求）     |
| 3   | `constructor(message, code)`           | 构造函数，接收参数                  |
| 4   | `super(message)`                       | 调用父类构造函数（必须先于 `this`） |
| 5   | `this.code = code`                     | 设置自定义属性                      |

**为什么必须 `super()`**:

```tsx
// ❌ 错误：在 super() 之前使用 this
constructor(message: string, code: number) {
  this.code = code  // 报错！
  super(message)
}

// ✅ 正确：先调用 super()
constructor(message: string, code: number) {
  super(message)   // 初始化父类
  this.code = code // 然后才能用 this
}
```

```tsx

error.code // 1 - 自定义属性

error instanceof GeolocationError // true
```

**代码示例**:

### 问题 #26: constructor 和 super() 的作用

**首次记录**: 2026-03-15 03:35
**最近记录**: 2026-03-15 03:35
**记录次数**: 1

**原始问题**:

> constructor有什么作用?这个是固定的函数吗？已经为什么一定要super(message)？有什么作用？

**知识点**: TypeScript / 类 / 继承

**核心解答**:

- **constructor**：类的初始化函数，名字固定，`new` 时自动执行
- **super()**：调用父类的 constructor，子类必须先初始化父类才能使用 `this`

**深入理解**:

```tsx
// constructor 是"初始化函数"
class Person {
  name: string
  constructor(name: string) {
    // 名字固定为 constructor
    this.name = name // 初始化属性
  }
}

const p = new Person('张三') // 自动调用 constructor
console.log(p.name) // '张三'
```

**为什么必须 super()？**

```
类比：
父类 Error = "毛坯房"（有 message 属性框架）
子类 = "精装修房"（在毛坯房上加 code）

super(message) = 先盖毛坯房
this.code = code = 然后装修

没有毛坯房，就没法装修！
```

**执行顺序**:

```tsx
class GeolocationError extends Error {
  constructor(message: string, code: number) {
    super(message)
    // ↑ 相当于：Error.call(this, message)
    // 设置 this.message = message
    // 设置 this.stack = 堆栈信息

    this.code = code  // 然后才能添加自己的属性
  }
}

// ❌ 错误：super() 之前用 this
constructor(message: string, code: number) {
  this.code = code   // 报错！'this' is not allowed before super()
  super(message)
}

// ✅ 正确：先 super() 再 this
constructor(message: string, code: number) {
  super(message)     // 先初始化父类
  this.code = code   // 再操作 this
}
```

**规则总结**:

| 情况         | 需要 super()? | this 可用时机    |
| ------------ | ------------- | ---------------- |
| 不继承任何类 | ❌ 不需要     | constructor 开头 |
| 继承父类     | ✅ 必须调用   | super() 之后     |

**代码示例**:

```tsx
// 不继承 — 不需要 super
class Person {
  name: string
  constructor(name: string) {
    this.name = name // ✅ 直接用 this
  }
}

// 继承 — 必须先 super
class GeolocationError extends Error {
  code: number
  constructor(message: string, code: number) {
    super(message) // ✅ 必须先调用
    this.code = code // 然后 this 才可用
  }
}
```

**学习要点**: ⭐ constructor 名字固定，`new` 时自动执行；继承类必须先 `super()` 初始化父类；`super()` 之后才能用 `this`

### 问题 #28: resolve 的作用 & fetch vs getCurrentPosition

**首次记录**: 2026-03-15 04:00
**最近记录**: 2026-03-15 04:00
**记录次数**: 1

**原始问题**:

> resolve有什么作用?就只是单纯的返回数据？以及fetch(url).then(a => a.json())和getCurrentPosition有什么区别？

**知识点**: JavaScript / Promise / 异步编程

**核心解答**:

- **resolve**：把 Promise 状态变成"成功"，传递结果给 `await`
- **fetch**：从**服务器**获取数据（需要网络）
- **getCurrentPosition**：从**设备**获取位置（GPS，不需要网络）

**深入理解**:

**Promise 三种状态**:

```
        pending（进行中）
           /        \
          /          \
    resolve()      reject()
        ↓              ↓
   fulfilled       rejected
   （已完成）       （已拒绝）

状态只能改变一次，不可逆
```

**resolve 和 await 的关系**:

```tsx
const getLocation = () => {
  return new Promise((resolve, reject) => {
    navigator.geolocation.getCurrentPosition(
      position => resolve(position), // ← 发出结果
      error => reject(error)
    )
  })
}

const position = await getLocation() // ← 接收结果
// resolve(position) 和 await 是"配对"的
```

**类比**:

```
resolve() = 快递员把包裹放进快递柜
await     = 你从快递柜取出包裹
Promise   = 快递柜（连接两端）
```

**fetch vs getCurrentPosition 对比**:

| 对比项   | fetch           | getCurrentPosition |
| -------- | --------------- | ------------------ |
| 数据来源 | 服务器          | 设备（GPS/WiFi）   |
| 需要网络 | ✅ 需要         | ❌ 不需要          |
| 返回类型 | Promise（原生） | Promise（封装后）  |
| 用途     | 获取服务器数据  | 获取用户位置       |

**代码示例**:

```tsx
// fetch — 从服务器获取数据
const weather = await fetch('https://api.weather.com').then(r => r.json())
// weather = { temp: 25, humidity: 60, ... }

// getCurrentPosition — 从设备获取位置
const position = await getCurrentPosition()
// position = { coords: { latitude: 39.9, longitude: 116.4 } }

// 典型场景：先定位，再查天气
const position = await getCurrentPosition()
const weather = await fetch(
  `https://api.weather.com?lat=${position.coords.latitude}&lon=${position.coords.longitude}`
)
```

**为什么都要封装成 Promise？**

```tsx
// ✅ 都是 Promise，可以用同一个流程
const location = await getCurrentPosition()
const weather = await fetch(url).then(r => r.json())

// ❌ 如果 getCurrentPosition 是回调风格，难以配合

  position => {
    /* 怎么传给 React Query？麻烦！*/
  },
  error => {
    /* 怎么处理错误？麻烦！*/
  }
)
```

**学习要点**: ⭐ resolve 把 Promise 变成成功状态，结果传给 await；fetch 从服务器取数据，getCurrentPosition 从设备取位置；都封装成 Promise 才能配合 React Query

### 问题 #29: await 与 Promise 的结果传递机制

**首次记录**: 2026-03-15 04:15
**最近记录**: 2026-03-15 04:15
**记录次数**: 1

**原始问题**:

> 有关结果传递给 await 解析的详细一点，比如await只是一个会在中间等待promise完成的中间变量是吗？

**知识点**: JavaScript / Promise / async-await

**核心解答**:
**await 不是"变量"，而是一个"暂停标记"**。它会暂停函数执行，等待 Promise 完成，然后"取出" resolve 传递的结果。

**深入理解**:

**await 的两个功能**:

| 功能     | 说明                                   |
| -------- | -------------------------------------- |
| **暂停** | 暂停 async 函数执行，等待 Promise 完成 |
| **取值** | 取出 resolve() 传递的结果              |

```tsx
const position = await getCurrentPosition()
//     ↑            ↑
//   取值          暂停
```

**resolve → await 的传递过程**:

```
Promise 内部                     调用者
    │                              │
    │  resolve(position)           │
    │       │                      │
    │       ▼                      │
    │  ┌─────────────┐             │
    │  │   Promise   │             │
    │  │  (快递柜)    │             │
    │  │  position   │ ◄── 存数据  │
    │  └─────────────┘             │
    │         │                    │
    │         ▼                    │
    │        await                 │
    │         │                    │
    │         ▼                    │
    │   const pos = position       │
    │      (取数据)                 │
```

**详细执行时间线**:

```tsx
async function example() {
  console.log('1. 开始')

  const result = await new Promise(resolve => {
    console.log('2. Promise 内部')
    setTimeout(() => {
      console.log('3. setTimeout 执行')
      resolve('数据') // ← 把 '数据' 放进 Promise
    }, 1000)
  })
  //                    ↑ await 在这里"卡住"等待

  console.log('4. 收到:', result) // result = '数据'
}

example()
console.log('5. 同步代码')

// 输出顺序：
// 1. 开始
// 2. Promise 内部
// 5. 同步代码      ← await 暂停期间，其他代码继续
// （等待 1 秒）
// 3. setTimeout 执行
// 4. 收到: 数据    ← await 取出 resolve 传的值
```

**为什么需要 await？**

```tsx
// ❌ 没有 await
const promise = getCurrentPosition()
console.log(promise) // Promise { <pending> }
// 只拿到 Promise 对象，没有数据！

// ✅ 有 await
const position = await getCurrentPosition()
console.log(position) // { coords: { latitude: 39.9 } }
// 拿到了真正的数据！
```

**Promise 内部如何存储结果（简化版）**:

```tsx
class MyPromise {
  #result = undefined // 隐藏变量，存储结果
  #state = 'pending'

  resolve(value) {
    this.#result = value // 存储结果
    this.#state = 'fulfilled'
  }
}

// await 会：
// 1. 检查 Promise 状态
// 2. 如果 pending，暂停等待
// 3. 如果 fulfilled，取出 #result
// 4. 把 #result 赋值给变量
```

**学习要点**: ⭐ await = 暂停 + 取值；`resolve(数据)` 存进 Promise；`await` 取出数据；传递路径：`resolve` → Promise 内部 → `await` → 变量

**原始问题**:

> 为什么可以把这个叫做一个自创的hook？什么是hook？hook一般要做什么事情？只是单纯的传递参数吗？其他API获取的数据也可以写成hook吗？

**知识点**: React / 自定义 Hook / 代码组织

**核心解答**:
**Hook 是 React 提供的一种机制，让你在函数组件中"钩入"状态和生命周期特性。自定义 Hook 本质上就是一个**以 `use` 开头的普通函数**，它内部可以调用其他 Hook。**

**判断自定义 Hook 的两个条件**:

| 条件                   | 说明                                   |
| ---------------------- | -------------------------------------- |
| 1. 函数名以 `use` 开头 | React 约定                             |
| 2. 内部调用了其他 Hook | `useState`、`useEffect`、`useQuery` 等 |

**深入理解**:

**为什么 `useUserLocation` 是自定义 Hook？**

```tsx
export const useUserLocation = () => {  // ✅ 以 use 开头
  return useQuery({...})                // ✅ 内部调用了 useQuery
}
// 两个条件都满足 → 就是自定义 Hook！
```

**Hook 能做什么**:

| 功能             | 示例                                   |
| ---------------- | -------------------------------------- |
| 封装 API 请求    | `useUser()`、`useWeather(lat, lon)`    |
| 封装复杂状态逻辑 | `useForm()`（表单验证）                |
| 封装浏览器 API   | `useLocalStorage()`、`useWindowSize()` |
| 封装订阅/事件    | `useOnlineStatus()`                    |

**不仅仅是传递参数**，Hook 可以封装复杂逻辑：

```tsx
// 示例：表单验证 Hook
export const useForm = <T>(initialValues: T) => {
  const [values, setValues] = useState(initialValues)
  const [errors, setErrors] = useState({})

  const handleChange = (field: keyof T, value: string) => {
    setValues(prev => ({ ...prev, [field]: value }))
    // 自动验证逻辑...
  }

  return { values, errors, handleChange }
}

// 示例：窗口大小 Hook
export const useWindowSize = () => {
  const [size, setSize] = useState({ width: 0, height: 0 })

  useEffect(() => {
    const handleResize = () => setSize({ width: window.innerWidth, height: window.innerHeight })
    window.addEventListener('resize', handleResize)
    return () => window.removeEventListener('resize', handleResize)
  }, [])

  return size
}
```

**其他 API 数据都可以写成 Hook**:

```tsx
// 天气
export const useWeather = (lat: number, lon: number) => {
  return useQuery({ queryKey: ['weather', lat, lon], queryFn: () => fetchWeather(lat, lon) })
}

// 用户信息
export const useUser = (userId: string) => {
  return useQuery({ queryKey: ['user', userId], queryFn: () => fetchUser(userId) })
}

// 搜索建议
export const useSearchSuggestions = (keyword: string) => {
  return useQuery({
    queryKey: ['suggestions', keyword],
    queryFn: () => fetchSuggestions(keyword),
    enabled: keyword.length > 0,
  })
}
```

**自定义 Hook 的好处**:

```tsx
// ❌ 不用 Hook：每个组件重复写
function ComponentA() {
  const { data, isLoading } = useQuery({...})  // 重复
}

// ✅ 用 Hook：简洁复用
function ComponentA() {
  const location = useUserLocation()  // 简洁！
}
```

| 好处 | 说明               |
| ---- | ------------------ |
| 复用 | 相同逻辑多处使用   |
| 简洁 | 组件代码更清晰     |
| 维护 | 修改一处，处处更新 |
| 测试 | Hook 可以单独测试  |

**学习要点**: ⭐ 自定义 Hook = `use` 开头 + 内部调用其他 Hook；用于封装复用逻辑；任何需要复用的逻辑都可以抽成 Hook

### 问题 #31: Zustand vs 自定义 Hook 的职责划分

**首次记录**: 2026-03-15 04:50
**最近记录**: 2026-03-15 04:50
**记录次数**: 1

**原始问题**:

> 如果每个 API 都写成 Hook，那 Zustand 用在哪？它们的作用好像重叠了？

**知识点**: React / 状态管理 / 架构设计

**核心解答**:
**它们解决的是不同类型的状态问题**：

- **React Query（自定义 Hook）**：管理**服务端状态**（API 数据）
- **Zustand**：管理**客户端状态**（用户选择、UI 状态、跨组件共享）

**深入理解**:

**状态类型划分**:

| 状态类型   | 管理方案    | 示例                           |
| ---------- | ----------- | ------------------------------ |
| 服务端状态 | React Query | 天气数据、用户信息、API 数据   |
| 客户端状态 | Zustand     | 用户选择的位置、主题、登录状态 |

**为什么不能混用？**

```tsx
// ❌ 错误：用 Zustand 管理 API 数据
const useWeatherStore = create(set => ({
  weather: null,
  isLoading: false,
  fetchWeather: async (lat, lon) => {
    set({ isLoading: true })
    const data = await fetchWeather(lat, lon) // 手动处理
    set({ weather: data, isLoading: false })
  },
}))
// 问题：没缓存、没刷新、没去重、代码多

// ✅ 正确：用 React Query
const { data: weather, isLoading } = useWeather(lat, lon)
// 自动缓存、刷新、去重，一行搞定
```

**判断标准（三个问题）**:

| 问题                           | 是 → Zustand | 否 → React Query           |
| ------------------------------ | ------------ | -------------------------- |
| 数据来自服务器还是用户操作？   | 用户操作     | 服务器                     |
| 需要跨组件共享吗？             | 是           | React Query 自动共享       |
| 需要持久化到 localStorage 吗？ | 是           | 不需要（服务器数据会过期） |

**完整示例**:

```tsx
// ========== Zustand：客户端状态 ==========
// 用户选择的位置（跨组件共享）
export const useLocationStore = create<LocationStore>(set => ({
  location: { lat: 39.9, lon: 116.4 },
  setLocation: location => set({ location }),
}))

// ========== Hook：服务端状态 ==========
// 天气数据（来自 API）
export const useWeather = (lat: number, lon: number) => {
  return useQuery({
    queryKey: ['weather', lat, lon],
    queryFn: () => fetchWeather(lat, lon),
  })
}

// ========== 组件使用 ==========
function WeatherApp() {
  const { location, setLocation } = useLocationStore() // 客户端状态
  const { data: weather } = useWeather(location.lat, location.lon) // 服务端状态

  return <WeatherCard data={weather} />
}
```

**决策表**:

| 场景               | 解决方案          |
| ------------------ | ----------------- |
| API 数据           | React Query       |
| 用户操作产生的状态 | Zustand           |
| 跨组件共享的状态   | Zustand           |
| 需要持久化的状态   | Zustand + persist |

**学习要点**: ⭐ React Query 管**服务端状态**（API 数据），Zustand 管**客户端状态**（用户选择）；数据来自服务器 → React Query，数据来自用户 → Zustand；不要把 API 数据存到 Zustand

### 问题 #32: React Query enabled 选项的作用

**首次记录**: 2026-03-15 05:15
**最近记录**: 2026-03-15 05:15
**记录次数**: 1

**原始问题**:

> 为什么一旦少了 `enabled: lon !== null && lat !== null`，刷新页面时就不会渲染城市数据到卡片上？

**知识点**: React Query / enabled 选项

**核心解答**:
`enabled` 控制**查询是否启用**。当 `enabled: false` 时查询不执行；当 `enabled` 从 `false` 变成 `true` 时，查询会**自动触发**。

**深入理解**:

**执行时间线对比**:

```
没有 enabled 时：
─────────────────────────────────────
页面刷新 → 组件挂载
lon = null, lat = null（用户位置还在获取中）
    ↓
查询立即执行（没有 enabled 阻止）
queryKey: ['searchCity', null, null]
API: ?location=null,null → 错误！
    ↓
缓存了错误结果 ❌
    ↓
用户位置获取成功，lon = 116, lat = 39
但可能因为错误状态没有正确更新


有 enabled 时：
─────────────────────────────────────
页面刷新 → 组件挂载
lon = null, lat = null
enabled: null !== null → false
    ↓
查询等待，不执行 ✅
    ↓
用户位置获取成功，lon = 116, lat = 39
enabled: 116 !== null → true
    ↓
查询自动触发！✅
API: ?location=116,39 → 成功！
```

**enabled 的行为**:

| enabled 值          | 行为                                  |
| ------------------- | ------------------------------------- |
| `false`             | 查询**不执行**，返回 `status: 'idle'` |
| 从 `false` → `true` | 查询**自动触发**                      |
| `true`              | 正常执行查询                          |

**代码示例**:

```tsx
// 依赖其他数据的查询
const { data: userLocation } = useUserLocation()

const { data: weather } = useWeather(userLocation?.lon, userLocation?.lat)
// ↓ 如果没有 enabled，会用 null 立即查询

// 正确写法：等待条件满足
export const useWeather = (lon: number, lat: number) => {
  return useQuery({
    queryKey: ['weather', lon, lat],
    queryFn: () => fetchWeather(lon, lat),
    enabled: lon != null && lat != null, // ← 等待条件满足
  })
}
```

**类比理解**:

```
enabled 就像一个"开关"

没有 enabled：水龙头一直开着，水还没来就开始放，放出来的是空气（错误）
有 enabled：水龙头先关着，等水来了（条件满足）自动打开，放出来的是水（正确数据）
```

**最佳实践**:

| 场景         | enabled 设置                   |
| ------------ | ------------------------------ |
| 依赖其他数据 | `enabled: !!dependentData`     |
| 条件查询     | `enabled: keyword.length >= 2` |
| 手动触发     | `enabled: false` + `refetch()` |

**学习要点**: ⭐ `enabled: false` 让查询等待；条件满足后自动触发；依赖其他数据时必须用 enabled 等待条件满足，否则会用错误参数立即查询

### 问题 #33: React Query 的响应式机制

**首次记录**: 2026-03-15 05:30
**最近记录**: 2026-03-15 05:30
**记录次数**: 1

**原始问题**:

> enabled 类似 await 吗？数据只传入一次？还是参数变化后重新执行？我只是单纯没更新界面？

**知识点**: React Query / 响应式机制

**核心解答**:
**不是 await，而是"响应式条件"**。数据会**多次传入**（每次渲染），React Query 会**自动检测变化并重新查询**。

**深入理解**:

**组件渲染多次**:

```tsx
function MyComponent() {
  const { data: userLocation } = useUserLocation()
  const { lon, lat } = userLocation ?? { lon: null, lat: null }
  const { data: cityWeather } = useCityWeatherData(lon, lat)

  console.log('渲染', { lon, lat })
  return <WeatherCard data={cityWeather} />
}

// 控制台输出：
// 渲染 { lon: null, lat: null }      ← 第1次渲染
// 渲染 { lon: 116, lat: 39 }         ← 第2次渲染（位置获取成功）
```

**完整时间线**:

```
第1次渲染：lon=null, lat=null
    useQuery 执行
    queryKey: ['searchCity', null, null]
    enabled: false
    → 查询等待，不请求

第2次渲染：lon=116, lat=39  ← 参数变了！
    useQuery 再次执行
    queryKey: ['searchCity', 116, 39]  ← key 变了！
    enabled: true  ← 条件满足了！
    → 自动发起 API 请求
```

**React Query 响应式原理**:

```tsx
useQuery({
  queryKey: ['searchCity', lon, lat], // ← 依赖数组
  enabled: lon !== null,
})

// React Query 会：
// 1. 每次渲染都执行 useQuery
// 2. 检测 queryKey 是否变化
// 3. 检测 enabled 是否变化
// 4. 满足条件时自动触发新查询
```

**关键区别**:

| 概念         | 说明                                   |
| ------------ | -------------------------------------- |
| enabled      | **声明式条件**，不是 await（阻塞等待） |
| 数据传入次数 | **多次**，每次渲染都传入新值           |
| 触发重新查询 | queryKey 变化 + enabled 为 true        |
| 不是界面问题 | 是 **API 请求时机** 的问题             |

**类比**:

```
enabled ≠ await（阻塞等待，代码停在那）
enabled = 条件开关（声明式，React Query 自动检测变化）
```

**学习要点**: ⭐ 数据多次传入（每次渲染）；enabled 是声明式条件非阻塞等待；React Query 自动检测 queryKey 和 enabled 变化；变化时自动触发新查询

### 问题 #34: React Query 的缓存机制

**首次记录**: 2026-03-15 05:45
**最近记录**: 2026-03-15 05:45
**记录次数**: 1

**原始问题**:

> queryKey 中的 lon/lat 是"上一次"传参的内容吗？即使单独开一个 Hook，useQuery 也记录着之前搜索的地点？

**知识点**: React Query / 缓存机制

**核心解答**:
**queryKey 是"当前"的值，不是"上一次"的值**。React Query 会为**每个不同的 queryKey 单独缓存**，多个地点的数据可以同时存在于缓存中。

**深入理解**:

**缓存结构（类似 Map）**:

```
React Query 缓存
─────────────────────────────────────────────
queryKey                     | 缓存数据
─────────────────────────────────────────────
['searchCity', 116, 39]     | { city: '北京' }
['searchCity', 121, 31]     | { city: '上海' }
['searchCity', 113, 23]     | { city: '广州' }
['weather', 116, 39]        | { temp: 25 }
─────────────────────────────────────────────

所有查询结果同时存在！切换回已缓存的 key 直接返回，不重新请求
```

**queryKey 始终是当前值**:

```tsx
function MyComponent() {
  const [lon, setLon] = useState(116)

  const { data } = useQuery({
    queryKey: ['searchCity', lon], // ← 始终是当前渲染时的 lon
  })

  // 第1次渲染：lon=116 → queryKey: ['searchCity', 116]
  // setLon(121) 后
  // 第2次渲染：lon=121 → queryKey: ['searchCity', 121]
  // 不是"上一次"的值，是"当前"的值！
}
```

**缓存生命周期**:

```tsx
const { data } = useQuery({
  queryKey: ['searchCity', lon, lat],
  staleTime: 5 * 60 * 1000, // 5分钟内数据是"新鲜的"
  gcTime: 10 * 60 * 1000, // 10分钟后清除缓存
})

// 时间线：
// 请求 → 缓存数据 → 5分钟内（新鲜期）直接返回缓存
//                → 超过5分钟（过期期）后台刷新
//                → 超过10分钟（清除期）缓存删除
```

**实际场景**:

```tsx
// 用户操作：
// 1. 搜索"北京" → queryKey: ['searchCity', '北京'] → 请求 → 缓存
// 2. 搜索"上海" → queryKey: ['searchCity', '上海'] → 请求 → 缓存
// 3. 再次搜索"北京" → queryKey: ['searchCity', '北京']
//    → 发现已缓存！直接返回，不请求 API ✅
```

**回答总结**:

| 问题                      | 答案                             |
| ------------------------- | -------------------------------- |
| queryKey 是"上一次"的值？ | **不是**，是**当前渲染时**的值   |
| 记住之前搜索的地点？      | **是的**，每个 queryKey 单独缓存 |
| 多个地点同时存在缓存？    | **是的**，缓存是 Map 结构        |

**学习要点**: ⭐ queryKey 是当前值非上次值；每个 key 单独缓存；多个地点可同时存在；重新访问已缓存 key 直接返回不请求

### 问题 #35: React Query 全局缓存池

**首次记录**: 2026-03-15 05:55
**最近记录**: 2026-03-15 05:55
**记录次数**: 1

**原始问题**:

> 也就是说 useQuery 是共用一个缓存池？

**知识点**: React Query / QueryClient / 缓存架构

**核心解答**:
**是的！所有 `useQuery` 共用一个全局缓存池（QueryClient）。** 相同 queryKey 的查询共享数据，自动去重，跨组件同步。

**深入理解**:

**缓存池架构**:

```tsx
// 应用入口
const queryClient = new QueryClient()  // 全局缓存池

<QueryClientProvider client={queryClient}>
  <App />
</QueryClientProvider>
```

```
┌─────────────────────────────────────────┐
│            QueryClient（全局缓存池）       │
│  ┌───────────────────────────────────┐  │
│  │ ['weather', 116, 39] → { temp:25 }│  │
│  │ ['user', 1]         → { name:'张三' }│  │
│  └───────────────────────────────────┘  │
│              ↑ 共享                      │
│   ┌──────────┼──────────┐               │
│ ComponentA ComponentB ComponentC        │
│  useQuery   useQuery   useQuery         │
└─────────────────────────────────────────┘
```

**相同 queryKey = 共享数据**:

```tsx
function ComponentA() {
  const { data } = useQuery({ queryKey: ['weather', 116, 39] })
}

function ComponentB() {
  const { data } = useQuery({ queryKey: ['weather', 116, 39] }) // 相同 key
}

// ComponentA 请求一次
// ComponentB 直接用缓存，不重复请求
```

**三大好处**:

| 好处           | 说明                           |
| -------------- | ------------------------------ |
| **自动去重**   | 多个组件用相同 key，只请求一次 |
| **跨组件同步** | 一个组件更新，其他自动同步     |
| **预加载**     | 可提前缓存数据，秒开页面       |

**预加载示例**:

```tsx
// 列表页：鼠标悬停时预加载
;<div
  onMouseEnter={() => {
    queryClient.prefetchQuery({
      queryKey: ['weather', city.lon, city.lat],
      queryFn: () => fetchWeather(city.lon, city.lat),
    })
  }}
>
  {city.name}
</div>

// 详情页：数据已缓存，秒开
const { data } = useQuery({ queryKey: ['weather', lon, lat] })
```

**学习要点**: ⭐ 所有 useQuery 共享全局 QueryClient；相同 queryKey 共享数据、自动去重、跨组件同步；可预加载提升体验

### 问题 #36: React Query 防抖设计

**首次记录**: 2026-03-15 08:00
**最近记录**: 2026-03-15 08:00
**记录次数**: 1

**原始问题**:

> 对于我的 useCityLocation 我应该如何设计防抖？

**知识点**: React / React Query / 防抖（Debounce）

**核心解答**:

**防抖（Debounce）**：用户停止输入一段时间后才执行操作。避免用户每敲一个字就发一次请求，浪费资源。

**实现位置**：在传入 `useQuery` 之前对搜索词进行防抖处理。

**深入理解**:

```tsx
// 当前代码问题：
const data = useQuery({
  queryKey: ['cityLocation', cityName], // cityName 每次变化都触发新请求
  enabled: cityName.length > 1, // 只是控制"何时启用"，不是防抖
})
```

**方案一：useDeferredValue（React 18+ 内置）**

```tsx
import { useDeferredValue } from 'react'

export const useCityLocation = (cityName: string) => {
  const debouncedCityName = useDeferredValue(cityName) // React 自动延迟更新

  const data = useQuery({
    queryKey: ['cityLocation', debouncedCityName],
    queryFn: async () => fetch(url + debouncedCityName),
    enabled: debouncedCityName.length > 1,
  })
  return data
}
```

优点：React 内置，无需额外依赖
缺点：不能自定义延迟时间

**方案二：自定义 useDebouncedValue Hook（推荐）**

```tsx
// hooks/useDebouncedValue.ts
import { useState, useEffect } from 'react'

export const useDebouncedValue = <T>(value: T, delay: number = 500): T => {
  const [debouncedValue, setDebouncedValue] = useState(value)

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebouncedValue(value)
    }, delay)

    return () => clearTimeout(timer) // value 变化时取消之前的定时器
  }, [value, delay])

  return debouncedValue
}
```

```tsx
// useCityLocation.tsx
import { useDebouncedValue } from './useDebouncedValue'

export const useCityLocation = (cityName: string) => {
  const debouncedCityName = useDebouncedValue(cityName, 500) // 500ms 防抖

  const data = useQuery({
    queryKey: ['cityLocation', debouncedCityName],
    queryFn: async () => fetch(url + debouncedCityName),
    enabled: debouncedCityName.length > 1,
  })
  return data
}
```

**效果对比**:

```
用户输入：北 → 北京 → 北京市

【无防抖】输入"北京"→请求，输入"北京市"→请求，共 2 次

【有防抖】输入"北"→等待，输入"北京"→重新等待，输入"北京市"→重新等待，
停止输入 500ms 后→发送请求，共 1 次 ✅
```

**方案选择**:

| 方案               | 延迟时间   | 依赖      | 适用场景     |
| ------------------ | ---------- | --------- | ------------ |
| `useDeferredValue` | React 自动 | React 18+ | 简单场景     |
| 自定义 Hook        | 可自定义   | 无        | 需要精确控制 |
| lodash.debounce    | 可自定义   | lodash    | 复杂场景     |

**学习要点**: ⭐ 防抖 = 用户停止输入一段时间后才执行；用 `setTimeout` + `clearTimeout` 实现；搜索场景推荐 300-500ms；在传入 `useQuery` 前对搜索词防抖

### 问题 #37: useEffect 清理机制与 setTimeout

**首次记录**: 2026-03-15 08:30
**最近记录**: 2026-03-15 08:30
**记录次数**: 1

**原始问题**:

> 为什么 useEffect 要监听 delay？setTimeout 会返回什么？如果 setTimeout 是异步执行的，那不是在执行前就 return 了吗？那 useEffect return 了什么？为什么 return 后面跟着的是 clearTimeout？它有什么作用？

**知识点**: React / useEffect / setTimeout

**核心解答**:

**useEffect 清理函数**：当组件卸载或依赖变化时，React 会执行上一次 effect 返回的清理函数。这是处理副作用的"打扫战场"机制。

**深入理解**:

**问题 1：为什么监听 delay？**

`delay` 决定了定时器的等待时间。如果 `delay` 变化，需要用新延迟重新设置定时器。不监听会导致定时器使用旧值。

**问题 2：setTimeout 返回什么？**

```tsx
const timer = setTimeout(() => {}, 1000)
console.log(timer) // 数字，如 1、2、3...
```

返回 **timer ID（数字）**，用于标识和取消这个定时器。

**问题 3：setTimeout 是异步的，不是执行前就 return 了吗？**

关键理解：`return () => clearTimeout(timer)` 返回的是**函数本身**，不是函数的执行结果！

```tsx
useEffect(() => {
  const timer = setTimeout(...)  // 第1步：创建定时器
  return () => clearTimeout(timer)  // 第2步：返回函数（还没执行！）
  // 第3步：稍后 React 调用这个函数
}, [value])
```

**类比**：

```tsx
// 就像你给朋友留了一把钥匙
useEffect(() => {
  const key = 开门
  return function 锁门() {
    // 这不是现在执行，而是"离开时执行"
    用钥匙锁门
  }
})
```

**问题 4 & 5：clearTimeout 的作用？**

**取消定时器，阻止回调执行**。这是防抖的核心逻辑！

```
【无 clearTimeout】
输入"北"  → timer=1
输入"北京"→ timer=2
输入"北京市"→ timer=3
500ms 后 timer=1,2,3 都触发 → 状态更新 3 次 ❌

【有 clearTimeout】
输入"北"  → timer=1
输入"北京"→ clearTimeout(1) 取消 → timer=2
输入"北京市"→ clearTimeout(2) 取消 → timer=3
500ms 后 timer=3 触发 → 状态更新 1 次 ✅
```

**完整流程**:

```
用户输入 value="北" → 创建 timer=1 → 返回清理函数
用户输入 value="北京" → React 调用清理函数 clearTimeout(1) → 创建 timer=2
用户停止输入 → 500ms 后 timer=2 回调执行 ✅
```

**关键概念表**:

| 概念               | 说明                                |
| ------------------ | ----------------------------------- |
| `setTimeout` 返回  | timer ID（数字），用于取消定时器    |
| `useEffect` return | 返回**清理函数**，不是立即执行      |
| 清理函数执行时机   | 依赖变化或组件卸载时，由 React 调用 |
| `clearTimeout`     | 取消定时器，阻止回调执行            |
| 防抖原理           | 每次输入取消旧定时器+创建新定时器   |

**学习要点**: ⭐ `return () => clearTimeout(timer)` 意思是"下次变化时，先把上次的定时器取消了"；清理函数由 React 在适当时机调用，不是立即执行

### 问题 #38: 函数定义 vs 函数调用 & Timer ID 原理

**首次记录**: 2026-03-15 09:00
**最近记录**: 2026-03-15 09:00
**记录次数**: 1

**原始问题**:

> 为什么 clearTimeout 不会立刻执行？而是在下一次执行？以及计时器的 ID 是如何计算的？是隔一段时间换一个 ID 还是在创建的时候就固定 ID 了？

**知识点**: JavaScript / 函数 / setTimeout

**核心解答**:

**函数定义 ≠ 函数调用**：`return () => clearTimeout(timer)` 返回的是函数本身（定义），不是执行结果。只有加 `()` 才会执行。

**Timer ID 在创建时固定**：`setTimeout` 调用时立即分配 ID，由浏览器内部计数器生成，创建后不会变化。

**深入理解**:

**问题 1：为什么 clearTimeout 不立刻执行？**

```tsx
// ❌ 函数调用 — 立刻执行
return clearTimeout(timer)

// ✅ 函数定义 — 返回函数本身，不执行
return () => clearTimeout(timer)
```

**类比**：

```tsx
// 菜谱 vs 做菜
const 菜谱 = () => 炒番茄炒蛋() // 菜谱（定义）
菜谱() // 做菜（调用）

console.log(菜谱) // 打印函数本身
console.log(菜谱()) // 执行函数，打印结果
```

**执行时间线**：

```
useEffect 执行 → setTimeout 创建 timer=1 → return () => clearTimeout(1)
                                                     ↑
                                              返回函数定义，不执行
React 保存这个函数

value 变化 → React 调用保存的函数 → clearTimeout(1) 执行 ✅
```

**问题 2：Timer ID 如何计算？**

**答案：创建时立即分配，固定不变。**

```tsx
const t1 = setTimeout(() => {}, 1000) // t1 = 1，立即返回
const t2 = setTimeout(() => {}, 1000) // t2 = 2，立即返回
const t3 = setTimeout(() => {}, 1000) // t3 = 3，立即返回

// ID 在创建时分配，不是根据时间
// 即使 t1 回调还没执行，t2 已经是 2 了
```

**浏览器内部机制（简化）**：

```tsx
let timerCounter = 0

function setTimeout(callback, delay) {
  timerCounter++ // 计数器 +1
  const id = timerCounter // 分配 ID
  timerMap.set(id, { callback, executeTime: Date.now() + delay })
  return id // 立即返回
}

function clearTimeout(id) {
  timerMap.delete(id) // 从映射表删除
}
```

**关键区别**：

| 概念         | 说明                     | 示例                           |
| ------------ | ------------------------ | ------------------------------ |
| 函数定义     | 返回函数本身             | `return () => fn()`            |
| 函数调用     | 立刻执行                 | `return fn()`                  |
| Timer ID     | 创建时立即分配，固定不变 | `setTimeout()` 返回 1, 2, 3... |
| clearTimeout | 从映射表删除，ID 不变    | `clearTimeout(1)` 后 ID 还是 1 |

**学习要点**: ⭐ `() => fn()` 是定义（不执行），`fn()` 是调用（立即执行）；Timer ID 创建时立即分配，固定不变，不会复用

### 问题 #39: useEffect 清理函数的执行时机

**首次记录**: 2026-03-15 09:30
**最近记录**: 2026-03-15 09:30
**记录次数**: 1

**原始问题**:

> 为什么监听的属性变化了就会执行 return 返回的函数，这有些反直觉不是吗？

**知识点**: React / useEffect / 生命周期

**核心解答**:

**useEffect 的设计哲学**：每次新 effect 执行前，先清理上一次的 effect。就像"离开房间前要打扫，然后才能进入新房间"。

**深入理解**:

**useEffect 生命周期**：

```
1. 组件挂载 → 执行 effect
2. 依赖变化 → 先执行清理函数 → 再执行新的 effect
3. 组件卸载 → 执行清理函数
```

**为什么这样设计？**

清理函数的作用是"打扫战场"——取消旧的定时器、移除事件监听器、取消网络请求。如果在开始新 effect 前不清理旧的，就会出问题：

```tsx
// 无清理机制
useEffect(() => {
  setTimeout(() => fetchData(value), 500)
}, [value])
// value 从 A→B → 发送两个请求，A 的结果可能覆盖 B ❌

// 有清理机制
useEffect(() => {
  const timer = setTimeout(() => fetchData(value), 500)
  return () => clearTimeout(timer) // 取消旧的
}, [value])
// value 从 A→B → 取消 A 的请求，只发送 B ✅
```

**类比：住酒店**

```
入住 → 拿钥匙（effect 执行）
换房 → 先退旧钥匙（清理函数）→ 再拿新钥匙（新 effect）
退房 → 退还钥匙（卸载时清理）
```

**React 内部实现（简化）**：

```tsx
let previousCleanup = null

function runEffect(effect) {
  if (previousCleanup) {
    previousCleanup() // 先清理旧的 ← 这就是为什么清理函数会执行！
  }
  previousCleanup = effect() // 执行新 effect，保存清理函数
}
```

**执行时间线**：

```
value=A → effect 执行，打印 "effect: A"，保存 cleanup_A
value=B → cleanup_A() 执行，打印 "cleanup: A"
          → effect 执行，打印 "effect: B"，保存 cleanup_B
value=C → cleanup_B() 执行，打印 "cleanup: B"
          → effect 执行，打印 "effect: C"，保存 cleanup_C
卸载    → cleanup_C() 执行，打印 "cleanup: C"
```

**为什么"反直觉"？**

习惯的直觉：`return 函数` → 函数执行完就结束了
React 的设计：`return 函数` → React 保存它 → 适当时机调用

这是 React 的"声明式"思想：你告诉 React "如果要清理，请执行这个函数"，React 决定何时调用。

**学习要点**: ⭐ useEffect 清理时机：每次新 effect 执行前先清理旧的；设计目的：防止旧副作用干扰新的；类比：换房间前先退旧钥匙；声明式：你提供清理函数，React 决定何时调用

**精确理解**:

```tsx
useEffect(() => {
  // effect 开始
  return () => {
    // effect 结束时调用
  }
  //      ↑ 告诉 React："这个 effect 结束时，请执行这个清理函数"
}, [value])
```

| 场景     | "结束"的含义                     |
| -------- | -------------------------------- |
| 依赖变化 | 新 effect 开始前，旧 effect 结束 |
| 组件卸载 | 组件从 DOM 移除前                |

**一句话总结**：`return 清理函数` 就是告诉 React "这个 effect 结束时请帮我打扫战场"
