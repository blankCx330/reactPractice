### 问题 #5: 经纬度默认值搞反导致API请求失败

**首次记录**: 2026-03-12 00:00
**最近记录**: 2026-03-12 00:00
**记录次数**: 1

**原始问题**:

> 为什么会出现这个错误？(API返回400 Bad Request)

**相关代码**:

```tsx
const [lon, setLon] = useState(userLocation?.lon ?? 39) // 错误！39是纬度
const [lat, setLat] = useState(userLocation?.lat ?? 116) // 错误！116是经度
```

**问题分析**:
经纬度默认值搞反了。北京坐标是经度116、纬度39，但代码把39赋给了lon，116赋给了lat。导致API请求变成`location=39,116`（错误的格式），服务器返回400错误。

**正确做法**:

```tsx
const [lon, setLon] = useState(userLocation?.lon ?? 116) // 经度
const [lat, setLat] = useState(userLocation?.lat ?? 39) // 纬度
```

**学习要点**: ⭐ 经度范围-180~180，纬度范围-90~90。API格式`location=经度,纬度`。北京是经度116、纬度39

**问题分析**:

**学习要点**: ⭐ `.map()`返回的元素必须添加`key`属性，通常用数据的唯一标识（如id、日期等）

### 问题 #7: 在 queryFn 内部使用 isLoading/isError

**首次记录**: 2026-03-14 15:00
**最近记录**: 2026-03-14 16:00
**记录次数**: 2

**原始问题**:

> 为什么不能在 queryFn 里写 if(isLoading)？

**历史问题**:

> - [2026-03-14 16:00] 那为什么这里又有可以判断状态？queryFn 里写 if(isError) 和 if(isLoading)

**相关代码**:

```tsx
const { isLoading, data } = useQuery({
  queryFn: async () => {
    const position = await getLocation()
    if (isLoading) {
      // ❌ 错误！isLoading 还没定义
      return { lat: position.coords.latitude, lon: position.coords.longitude }
    }
  },
})
```

**问题分析**:
`isLoading`/`isError` 是 `useQuery` **返回**的状态，在 `queryFn` 执行时还不存在。执行顺序是：调用 useQuery → 执行 queryFn → 返回 { isLoading, ... }。在 queryFn 里访问这些状态相当于在变量定义之前使用它。

**重要**：用户看到的"可以判断状态"的代码**同样是错误的**，不会正常工作：

```tsx
queryFn: async () => {
  if (isError) {
    // ❌ 同样错误！isError 未定义
    return <ErrorMessage /> // ❌ queryFn 不应返回 JSX
  }
}
```

这段代码会报错，而且 queryFn 不应该返回组件。

**正确做法**:

```tsx
// queryFn 只负责获取数据，不需要判断状态
const useUserLocation = () => {
  return useQuery<UserLocation>({
    queryKey: ['userLocation'],
    queryFn: async () => {
      const position = await getLocation()
      return {
        lat: position.coords.latitude,
        lon: position.coords.longitude,
      }
    },
  })
}

// 在组件中使用返回的状态
const { data, isLoading } = useUserLocation()
if (isLoading) {
  return <LoadingSpinner />
}
```

**学习要点**: ⭐ `queryFn` 是"干活的人"，`isLoading` 是"干完活后的报告"。干活时报告还没写出来，所以不能在 queryFn 里用 isLoading

### 问题 #4: URL查询参数格式错误

**首次记录**: 2026-03-11 10:30
**最近记录**: 2026-03-11 10:30
**记录次数**: 1

**原始问题**:

> 为什么这段代码无法返回数据？

**相关代码**:

```tsx
const { data: sevenWeather } = useQuery({
  queryKey: ['sevenWeather', lon, lat],
  queryFn: async () => {
    const url = `https://${apiHost}/v7/weather/7d?location=${lon},${lat}/key=${apiKey}`
    return fetch(url).then(res => res.json())
  },
})
```

**问题分析**:
URL 中用 `/key=` 连接查询参数，但 `/` 是路径分隔符，不是查询参数连接符。服务器会把 `/key=xxx` 当成路径的一部分，导致 API 返回错误。

**正确做法**:

```tsx
// 修复：把 /key= 改成 &key=
const url = `https://${apiHost}/v7/weather/7d?location=${lon},${lat}&key=${apiKey}`
```

**学习要点**: ⭐ URL查询参数格式：`?参数1=值1&参数2=值2`。第一个参数用 `?`，后续参数用 `&`，永远不要用 `/`

### 问题 #3: 在查询前修改查询参数的逻辑错误

**首次记录**: 2026-03-10 17:30
**最近记录**: 2026-03-10 18:00
**记录次数**: 2

**原始问题**:

> [2026-03-10 17:30] 为什么我的代码这么修改了还是无法正确的搜索？
> [2026-03-10 18:00] 为什么搜索两次才能输出正确结果？

**相关代码**:

```tsx
const handleKeyDown = (event: React.KeyboardEvent<HTMLInputElement>) => {
  if (event.key === 'Enter') {
    setCityName(getCityLocation?.poi?.[0]?.adm2 ?? '') // 先修改参数
    refetchCityLocation() // 再查询
  }
}
```

**问题分析**:
在触发查询**之前**就修改了查询参数（cityName），导致查询使用的是错误（空）的值。这是典型的"执行顺序错误"：应该是"用当前值查询 → 处理结果"，而不是"修改值 → 用错误值查询"。

**正确做法**:

```tsx
const handleKeyDown = async (event: React.KeyboardEvent<HTMLInputElement>) => {
  if (event.key === 'Enter' && cityName.trim()) {
    // 用当前用户输入的 cityName 查询，不要修改它
    const result = await refetchCityLocation()

    if (result.data?.poi?.[0]) {
      const poi = result.data.poi[0]
      onLocationChange(poi.lon, poi.lat)
    }
  }
}
```

**学习要点**: ⭐ 不要在查询前修改查询参数，也不要在 `setState` 后立即使用该状态。`setState` 是异步的，立即使用会得到旧值。解决方案：直接用变量本身，不要绕状态

### 问题 #9: Zustand 类型定义与 set 用法错误

**首次记录**: 2026-03-15 01:00
**最近记录**: 2026-03-15 01:00
**记录次数**: 1

**原始问题**:

> 这个设置有什么问题？有没有什么更优雅的方法？

**相关代码**:

```tsx
type Store = {
  location: { Location: Location } // ❌ 类型定义错误
  setStore: () => void // ❌ 方法类型不匹配
}

export const LocationStore = create(set => ({
  location: { lon: 116, lat: 39 },
  setStore: (newLocation: Location) => set({ newLocation }), // ❌ set 更新错误
}))
```

**问题分析**:

1. `{Location:Location}` 定义的是"包含 Location 属性的对象"，而不是 Location 类型本身
2. 类型定义无参数，实现却有参数，类型不匹配
3. `set({newLocation})` 是属性简写，等于添加新字段而不是更新现有字段

**正确做法**:

```tsx
type Location = { lon: number; lat: number }

type Store = {
  location: Location // 直接用 Location 类型
  setLocation: (location: Location) => void // 类型要匹配
}

export const LocationStore = create<Store>(set => ({
  location: { lon: 116, lat: 39 },
  setLocation: newLocation => set({ location: newLocation }),
  //                              ↑ 指定要更新的字段名
}))
```

**更优雅的写法**（分开设置经纬度）:

```tsx
type Store = {
  location: Location
  setLocation: (location: Location) => void
  setLon: (lon: number) => void
  setLat: (lat: number) => void
}

export const LocationStore = create<Store>(set => ({
  location: { lon: 116, lat: 39 },
  setLocation: location => set({ location }),
  setLon: lon =>
    set(state => ({
      location: { ...state.location, lon }, // 部分更新
    })),
  setLat: lat =>
    set(state => ({
      location: { ...state.location, lat },
    })),
}))
```

**学习要点**: ⭐ 类型直接用 `Location` 不是 `{Location:Location}`；`set({ field: value })` 要指定字段名；部分更新用回调 `set(state => ({ ... }))`

### 问题 #11: 函数声明语法错误

**首次记录**: 2026-03-15 05:00
**最近记录**: 2026-03-15 05:00
**记录次数**: 1

**原始问题**:

> 缺少返回类型批注的"useCityWeatherData"隐式具有"any"返回类型

**相关代码**:

```tsx
export function useCityWeatherData(lon:number, lat:number) => {
//                                                    ↑↑↑ 多余的 =>
  return useQuery({...})
}
```

**问题分析**:
混用了 `function` 声明和箭头函数语法。`function` 声明不需要 `=>`，`=>` 是箭头函数的语法。

**正确做法**:

```tsx
// 方式一：function 声明（去掉 =>）
export function useCityWeatherData(lon: number, lat: number) {

export const useCityWeatherData = (lon: number, lat: number) => {

}
```

**语法对比**:

| 声明方式      | 语法                    | 需要 =>？ |
| ------------- | ----------------------- | --------- |
| function 声明 | `function name() {}`    | ❌ 不需要 |
| 箭头函数      | `const name = () => {}` | ✅ 需要   |

**学习要点**: ⭐ `function name() {}` 没有 `=>`，`const name = () => {}` 才有 `=>`

### 问题 #12: 在事件处理函数中调用 Hook

**首次记录**: 2026-03-15 06:10
**最近记录**: 2026-03-15 06:10
**记录次数**: 1

**原始问题**:

> 这两段代码会冲突吗？如果 input 进行输入，getCityLocation 会更新吗？

**相关代码**:

```tsx
const { data: getCityLocation } = useCityLocation(inputCityName)

const handleInputChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setInputCityName(e.target.value)
  useCityLocation(e.target.value) // ❌ 错误！
  setShowSuggestions(true)
}
```

**问题分析**:
在事件处理函数中调用 Hook 是**严重错误**。Hook 必须在组件顶层调用，不能在事件处理函数、循环、条件语句中调用。React Query 会自动响应状态变化，不需要手动调用。

**正确做法**:

```tsx
// 组件顶层调用 Hook
const { data: getCityLocation } = useCityLocation(inputCityName)

const handleInputChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setInputCityName(e.target.value) // 只更新状态
  setShowSuggestions(true)
  // 不需要手动调用 useCityLocation！
}

// 数据流：
// setInputCityName() → 状态更新 → 组件重渲染
// → useCityLocation(新值) → 自动重新查询 → getCityLocation 更新
```

**Hook 调用规则**:

| ❌ 错误位置  | ✅ 正确位置    |
| ------------ | -------------- |
| 事件处理函数 | 组件顶层       |
| 循环体内     | 函数体开头     |
| 条件语句内   | 所有 Hook 之前 |

**学习要点**: ⭐ Hook 只能在组件顶层调用；状态变化会自动触发 Hook 重新执行；不需要在事件处理函数中手动调用 Hook

### 问题 #13: queryKey 未包含动态参数导致查询不更新

**首次记录**: 2026-03-15 06:25
**最近记录**: 2026-03-15 06:25
**记录次数**: 1

**原始问题**:

> 删除了 handleInputChange 中的 useCityLocation，但选择列表不会在输入时正确更新

**相关代码**:

```tsx
// useCityLocation.tsx
export const useCityLocation = (cityName: string) => {
  const data = useQuery({
    queryKey: ['cityLocation'], // ❌ 没有包含 cityName！
    queryFn: async () => {
      const cityUrl = `...location=${cityName}...`
      return fetch(cityUrl).then(res => res.json())
    },
    enabled: cityName.length > 1,
  })
  return data
}
```

**问题分析**:
`queryKey` 是固定的 `['cityLocation']`，没有包含动态参数 `cityName`。当 `cityName` 变化时，React Query 认为是同一个查询，不会重新请求，直接返回缓存。

**正确做法**:

```tsx
export const useCityLocation = (cityName: string) => {
  const data = useQuery({
    queryKey: ['cityLocation', cityName], // ✅ 包含动态参数
    queryFn: async () => {
      const cityUrl = `...location=${cityName}...`
      return fetch(cityUrl).then(res => res.json())
    },
    enabled: cityName.length > 1,
  })
  return data
}
```

**修复后的数据流**:

```
输入"北京" → queryKey: ['cityLocation', '北京'] → 请求 API
输入"上海" → queryKey: ['cityLocation', '上海'] → queryKey 变化 → 重新请求 ✅
```

**学习要点**: ⭐ queryKey 必须包含所有动态参数；参数变化时 queryKey 也应该变化；否则 React Query 会认为是同一个查询，不重新请求

### 问题 #14: 搜索建议列表缺少空状态提示

**首次记录**: 2026-03-15 07:00
**最近记录**: 2026-03-15 07:00
**记录次数**: 1

**原始问题**:

> 发现当 getCityLocation 搜索不到城市的时候并没有给用户提示，这非常不好

**相关代码**:

```tsx
{
  showSuggestions && getCityLocation && (
    <div className="suggestions-list">
      {getCityLocation?.location?.map((data: Location) => (
        <div key={data.id} className="suggestion-item">
          {data.adm2} - {data.name} - {data.adm1}
        </div>
      ))}
    </div>
  )
}
```

**问题分析**:
条件渲染只处理了"有结果"的情况，忽略了"无结果"时的用户反馈。当 `location=[]` 时，`map` 返回空数组，div 内没有任何内容，用户不知道是"搜索不到"还是"出了问题"。

**不修复的后果**:
用户输入不存在的城市名，下拉框空空如也，用户困惑：是搜索不到？网络错误？还是正在搜索？可能反复尝试输入，体验极差。

**正确做法**:

```tsx
{showSuggestions && (
  <div className="suggestions-list">
    {/* 情况1：正在加载 */}
    {isFetching && (
      <div className="suggestion-item">搜索中...</div>
    )}

    {/* 情况2：搜索无结果 */}
    {!isFetching && getCityLocation?.location?.length === 0 && (
      <div className="suggestion-item">未找到匹配的城市</div>
    )}

    {/* 情况3：有结果 */}
    {!isFetching && getCityLocation?.location?.map((data: Location) => (
      <div key={data.id} className="suggestion-item" onClick={...}>
        {data.adm2} - {data.name} - {data.adm1}
      </div>
    ))}
  </div>
)}
```

**更优雅的方案（提取渲染函数）**:

```tsx
const renderSuggestions = () => {
  if (!showSuggestions) return null

  if (isFetching) return <div className="suggestion-item">搜索中...</div>

  if (!getCityLocation?.location?.length) {
    return <div className="suggestion-item">未找到匹配的城市</div>
  }

  return getCityLocation.location.map((data: Location) => (
    <div key={data.id} className="suggestion-item" onClick={...}>
      {data.adm2} - {data.name} - {data.adm1}
    </div>
  ))
}

// JSX 中使用
<div className="suggestions-list">{renderSuggestions()}</div>
```

**学习要点**: ⭐ 条件渲染三状态：加载中 → 无结果 → 有结果，缺一不可。用户体验核心是"让用户知道发生了什么"

### 问题 #15: 条件判断逻辑取反错误

**首次记录**: 2026-03-15 07:30
**最近记录**: 2026-03-15 07:45
**记录次数**: 2

**原始问题**:

> 为什么我在输入第二个字的时候页面直接全部清空了？

**历史问题**:

> - [2026-03-15 07:45] 我说的清空是整个页面都不渲染了，不单单指这个列表不渲染，为什么少一个!号会导致这么严重的后果？

**相关代码**:

```tsx
const suggestionsListText = () => {
  if (!showSuggestions) return null

  if (cityLocationIsLoading) return <div>搜索中...</div>

  if (getCityLocation?.location?.length) {  // ❌ 逻辑取反错误！
    return <div>未找到匹配城市</div>
  }

  return getCityLocation?.location.map(data => ...)
}
```

**问题分析**:

条件判断逻辑完全反了！`getCityLocation?.location?.length` 返回数组长度（数字），当长度 > 0 时为 truthy。所以：

- **有数据**（length=2）→ 条件为 true → 显示"未找到匹配城市" ❌
- **无数据**（length=0）→ 条件为 false → 继续执行 map ✅

**为什么会整个页面崩溃？**

缺少 `!` 不仅导致逻辑错误，还会引发 JavaScript 运行时错误：

```
API 返回 { location: null }（某些 API 无结果时返回 null）
    ↓
getCityLocation?.location?.length → null?.length → undefined
    ↓
undefined 是 falsy，不进入 if 块
    ↓
执行 getCityLocation?.location.map(...)
    ↓
location 是 null，调用 null.map() ❌
    ↓
TypeError: Cannot read property 'map' of null
    ↓
React 组件崩溃，整个页面空白！
```

**关键原因**：React 组件中任何 JavaScript 错误都会导致整个组件树崩溃。React 无法确定组件状态是否一致，只能放弃渲染。

**错误传播链**：

```
一个 ! 缺失 → 条件判断逻辑反转 → 错误的数据状态进入下一行 → null.map() 抛出 TypeError → React 捕获错误 → 整个页面空白
```

**正确做法**:

```tsx
const suggestionsListText = () => {
  if (!showSuggestions) return null

  if (cityLocationIsLoading) return <div>搜索中...</div>

  // ✅ 用 ! 取反：无数据时才显示"未找到"
  if (!getCityLocation?.location?.length) {
    return <div>未找到匹配城市</div>
  }

  // 有数据时正常显示列表
  return getCityLocation.location.map(data => (

    </div>
  ))
}
```

**判断规则表**:

| 场景                 | 正确写法                   | 错误写法                   |
| -------------------- | -------------------------- | -------------------------- |
| 数组**有**数据时执行 | `if (arr.length) { ... }`  | `if (!arr.length) { ... }` |
| 数组**无**数据时执行 | `if (!arr.length) { ... }` | `if (arr.length) { ... }`  |

**学习要点**: ⭐ 判断"无结果"用 `!array.length`，判断"有结果"用 `array.length`。口诀："判断什么就写什么，取反加感叹号"

**防御性编程原则**：

1. 永远假设数据可能是 `null`/`undefined`
2. 使用可选链 `?.` 防止访问错误
3. 在调用方法（如 `.map()`）前确保数据存在
4. 一个缺失的 `!` 可能导致整个页面崩溃，因为错误会沿代码传播

### 问题 #22: 在组件渲染过程中直接调用 setState

**首次记录**: 2026-03-17 17:30
**最近记录**: 2026-03-17 17:30
**记录次数**: 1

**原始问题**:

> 这个 setCityId 会在 getCityLocation 变化的时候再次指向吗？

**相关代码**:

```tsx
const { data: getCityLocation } = useCityLocation(inputCityName)

if (getCityLocation) setCityId(getCityLocation?.location[0].name) // ❌ 在渲染中直接调用 setState！
```

**问题分析**:

在组件函数体中直接调用 `setState` 是严重错误。渲染函数必须是**纯函数**，不能有副作用。这会导致无限循环：

```
渲染 → getCityLocation 有值 → setCityId() → 状态更新 → 重新渲染
→ getCityLocation 还是有值 → 再次 setCityId() → 无限循环 → 页面崩溃
```

**不修复的后果**:

- 无限循环，页面卡死
- React 报错：`Too many re-renders. React limits the number of renders to prevent an infinite loop.`

**正确做法**:

```tsx
const { data: getCityLocation } = useCityLocation(inputCityName)

// ✅ 使用 useEffect 监听数据变化
useEffect(() => {
  if (getCityLocation?.location?.[0]?.name) {
    setCityId(getCityLocation.location[0].name)
  }
}, [getCityLocation]) // 依赖 getCityLocation
```

**setState 调用位置对照表**：

| 位置         | 能否调用 setState | 原因                           |
| ------------ | ----------------- | ------------------------------ |
| 组件函数体   | ❌ 不能           | 渲染阶段，必须是纯函数         |
| useEffect    | ✅ 可以           | 渲染之后执行，副作用的安全位置 |
| 事件处理函数 | ✅ 可以           | 用户触发，不在渲染流程中       |

**学习要点**: ⭐ `setState` 只能在 `useEffect` 或事件处理函数中调用，绝不能在组件函数体中直接调用

### 问题 #42: React Component 泛型参数 <Props, State>

**首次记录**: 2026-03-16 10:45
**最近记录**: 2026-03-16 10:45
**记录次数**: 1

**原始问题**:

> `class ErrorBoundary extends Component<Props, State>` 这里的 `<Props, State>` 有什么作用？

**知识点**: TypeScript / React / 泛型

**核心解答**:
`<Props, State>` 是 TypeScript 泛型语法，告诉 React 的 `Component` 基类：这个组件接收什么类型的 props，维护什么类型的 state。

**深入理解**:

```tsx
// Component 内部定义（简化）
class Component<P, S> {
  props: P // Props 类型
  state: S // State 类型
}

// 使用示例
interface Props {
  children: React.ReactNode
  fallback?: React.ReactNode
}

interface State {
  hasError: boolean
  error: Error | null
}

class ErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  render() {
    // ✅ this.props 自动获得 Props 类型提示
    // ✅ this.state 自动获得 State 类型提示
    if (this.state.hasError) {
      return this.props.fallback || <div>出错了</div>
    }
    return this.props.children
  }
}
```

**类型安全效果**:
| 写法 | this.props 类型 | this.state 类型 |
|-----|----------------|----------------|
| `Component` | `any` ❌ | `{}` ❌ |
| `Component<Props>` | Props ✅ | `{}` |
| `Component<Props, State>` | Props ✅ | State ✅ |

**学习要点**: ⭐ `Component<Props, State>` 第一个参数是 props 类型，第二个是 state 类型。作用：获得完整的类型检查和智能提示

### 问题 #43: ErrorBoundary 生命周期方法与 ReactNode 类型

**原始问题**:

> getDerivedStateFromError 和 componentDidCatch 有什么作用？方法名必须这样拼写吗？返回类型写在 `:` 后面是否优雅？ReactNode 是什么类型？

两个方法都是 React 类组件的生命周期方法，用于捕获子组件错误。方法名必须完全匹配，React 通过名称识别。`: State` 是 TypeScript 返回类型标注。

**深入理解**:

```tsx
// getDerivedStateFromError - 更新 state，显示降级 UI
static getDerivedStateFromError(error: Error): State {
  // ^^^^^ 静态方法，无法访问 this
  //                       ^^^^^ 返回类型标注
  return { hasError: true, error }
}

// componentDidCatch - 记录日志、错误上报
componentDidCatch(error: Error, errorInfo: ErrorInfo): void {
  // ^^^^^^^^^^^^^ 实例方法，可以访问 this
  //                                          ^^^^^ 无返回值
  console.error('错误：', error)
  console.error('组件栈：', errorInfo.componentStack)
}
```

**两方法区别**:
| 方法 | 调用时机 | 能否访问 this | 返回值 | 用途 |
|-----|---------|--------------|-------|------|
| getDerivedStateFromError | 渲染阶段 | ❌ | State | 更新 state |
| componentDidCatch | commit 阶段 | ✅ | void | 日志上报 |

**ReactNode 类型定义**:

```tsx
type ReactNode =
  | ReactElement // <div />
  | string // "文本"
  | number // 123
  | boolean // true/false
  | null // null
  | undefined // undefined
  | ReactFragment // <>...</>
```

**学习要点**: ⭐ 生命周期方法名必须完全匹配；`: Type` 是 TS 返回类型标注，推荐写；ReactNode = 所有可渲染内容的联合类型

### 问题 #3: React状态更新的批处理机制

**首次记录**: 2026-03-10 17:00
**最近记录**: 2026-03-10 17:00
**记录次数**: 1

**原始问题**:

> 为什么我这样更改后 setSearchStatus(true) 然后又 setSearchStatus(false) 不起作用？

**知识点**: React / 状态管理

**核心解答**:
React 的状态更新是**异步批处理**的。连续调用多次 `setState` 时，React 会把它们合并成一次渲染，中间状态会丢失。

**深入理解**:

```tsx
setSearchStatus(true) // 请求1
setSearchStatus(false) // 请求2
// React 实际执行：直接变成 false，跳过了 true
```

这就像你要从1楼到3楼再到2楼，但电梯只在最终目的地停下——你不会在3楼停留。

**正确做法**:

```tsx
// ❌ 错误：中间状态会丢失
setStatus(true)
setStatus(false)

// ✅ 方式1：用 useEffect 监听状态变化
useEffect(() => {
  if (status) {
    // 执行操作
    setStatus(false) // 完成后重置
  }
}, [status])

// ✅ 方式2：用异步操作
await doSomething()
setStatus(false)

// ✅ 方式3：换一种不依赖状态切换的方案
```

**学习要点**: ⭐ 连续的 `setState` 会被批处理合并，中间状态不会触发渲染。需要用 `useEffect` 监听或 `await` 等待

```tsx
const str = '  北京  '

  // '   ' 是 truthy，会执行
  // 但用空格查询，查不到结果
}

// 有 trim() 的正确做法
if (input.trim()) {
  // '   '.trim() → ''
  // '' 是 falsy，不会执行
}
```

```tsx
// 表单验证
const handleSubmit = () => {

const handleSearch = () => {
```

自定义下拉列表 = 输入框 + 实时搜索 + 条件渲染的列表 + 点击处理。核心是控制列表的显示/隐藏和点击外部关闭。

**深入理解**:

1. **状态管理**: 输入值、是否显示列表
2. **实时搜索**: useQuery + enabled 条件
3. **显示控制**: 条件渲染 `{show && data && <列表 />}`

5. **选择处理**: 点击项目后更新状态、关闭列表

```tsx
const listRef = useRef(null)

  return () => document.removeEventListener('mousedown', handleClick)
}, [])

  setInput(item.name)
  setShowList(false)
  onLocationChange(item.lon, item.lat)

  <div>
    <input
      ref={inputRef}
      value={input}
      onChange={e => {
        setInput(e.target.value)
        setShowList(true)
      }}
      onFocus={() => setShowList(true)}
    />
    {showList && data?.length > 0 && (
      <div ref={listRef} className="suggestions">
        {data.map(item => (
          <div key={item.id} onClick={() => handleSelect(item)}>

      </div>
    )}
  </div>
)
```

```tsx
str.padStart(6, ' ') // '   abc'

str.padEnd(2, ' ') // 'abc' (不变)
```

```tsx
return <WeatherCard data={data} />
```

```tsx
// 两种等价写法

if (isLoading) {

return isLoading ? <LoadingSpinner /> : <Content data={data} />
```

**学习要点**: ⭐ 提前返回不是"不返回"，而是"根据条件返回不同内容"。`isLoading=true`返回加载动画，`isLoading=false`返回正常内容

```tsx
const myAsyncFunction = (): Promise<结果类型> => {

    }
  })
}
```

```tsx
      // 手动检测
      try {
        const position = await getLocation() // resolve 的值
      } catch (error) {
        // reject 的错误
      }
```

**学习要点**: ⭐ `resolve(值)`=成功，`reject(错误)`=失败；必须调用其一；这是包装回调 API 为 Promise 的标准模式；React Query 的 `isError`/`data` 检测 Promise 状态

```tsx
const lon = userLocation?.lon ?? 116
const lat = userLocation?.lat ?? 39
```

### 问题 #21: Zustand 缺少泛型导致类型为 unknown


```
create()         → 就像给 TypeScript 一个"盲盒"，它不知道里面是什么
```

```tsx
class GeolocationError extends Error {
  code: number
  constructor(message: string, code: number) {
    super(message)
    this.code = code
  }
}

const getCurrentPosition = (options = {}) => {
  return new Promise((resolve, reject) => {
    if (!navigator.geolocation) {
      reject(new GeolocationError('浏览器不支持定位功能', 0))
      return
    }
    navigator.geolocation.getCurrentPosition(
      resolve,
      error => {
        const messages = {
          1: '用户拒绝了定位请求',
          2: '无法获取位置信息',
          3: '定位超时',
        }
        reject(new GeolocationError(messages[error.code], error.code))
      },
      options
    )
  })
}

  return useQuery({
    queryKey: ['userLocation'],
    queryFn: async () => {
      const position = await getCurrentPosition(options)
      return { lat: position.coords.latitude, lon: position.coords.longitude }

}
```

### 问题 #25: TypeScript 自定义错误类

**首次记录**: 2026-03-15 03:30
**最近记录**: 2026-03-15 03:30
**记录次数**: 1

**核心解答**:

```tsx
throw new GeolocationError('用户拒绝定位', 1)
throw new GeolocationError('定位超时', 3)
//                          ↑ message    ↑ code

// 在 catch 中可以精确判断
try {
  await getLocation()
} catch (error) {
  if (error instanceof GeolocationError) {
    if (error.code === 1) {
      // 用户拒绝 - 提示用户开启权限
    } else if (error.code === 3) {
      // 超时 - 提示重试
    }
  }
}
```

**逐行解析**:

| 行  | 代码                                   | 作用                                |
| --- | -------------------------------------- | ----------------------------------- |
| 1   | `class GeolocationError extends Error` | 声明类并继承 `Error`                |
| 2   | `code: number`                         | 声明属性类型（TypeScript 要求）     |
| 3   | `constructor(message, code)`           | 构造函数，接收参数                  |
| 4   | `super(message)`                       | 调用父类构造函数（必须先于 `this`） |
| 5   | `this.code = code`                     | 设置自定义属性                      |

**为什么必须 `super()`**:

```tsx
// ❌ 错误：在 super() 之前使用 this
constructor(message: string, code: number) {
  this.code = code  // 报错！
  super(message)
}

// ✅ 正确：先调用 super()
constructor(message: string, code: number) {
  super(message)   // 初始化父类
  this.code = code // 然后才能用 this
}
```

```tsx

error.code // 1 - 自定义属性

error instanceof GeolocationError // true
```

### 问题 #26: constructor 和 super() 的作用

**首次记录**: 2026-03-15 03:35
**最近记录**: 2026-03-15 03:35
**记录次数**: 1

**原始问题**:

> constructor有什么作用?这个是固定的函数吗？已经为什么一定要super(message)？有什么作用？

**知识点**: TypeScript / 类 / 继承

**核心解答**:

- **constructor**：类的初始化函数，名字固定，`new` 时自动执行
- **super()**：调用父类的 constructor，子类必须先初始化父类才能使用 `this`

**深入理解**:

```tsx
// constructor 是"初始化函数"
class Person {
  name: string
  constructor(name: string) {
    // 名字固定为 constructor
    this.name = name // 初始化属性
  }
}

const p = new Person('张三') // 自动调用 constructor
console.log(p.name) // '张三'
```

**为什么必须 super()？**

```
类比：
父类 Error = "毛坯房"（有 message 属性框架）
子类 = "精装修房"（在毛坯房上加 code）

super(message) = 先盖毛坯房
this.code = code = 然后装修

没有毛坯房，就没法装修！
```

**执行顺序**:

```tsx
class GeolocationError extends Error {
  constructor(message: string, code: number) {
    super(message)
    // ↑ 相当于：Error.call(this, message)
    // 设置 this.message = message
    // 设置 this.stack = 堆栈信息

    this.code = code  // 然后才能添加自己的属性
  }
}

// ❌ 错误：super() 之前用 this
constructor(message: string, code: number) {
  this.code = code   // 报错！'this' is not allowed before super()
  super(message)
}

// ✅ 正确：先 super() 再 this
constructor(message: string, code: number) {
  super(message)     // 先初始化父类
  this.code = code   // 再操作 this
}
```

**规则总结**:

| 情况         | 需要 super()? | this 可用时机    |
| ------------ | ------------- | ---------------- |
| 不继承任何类 | ❌ 不需要     | constructor 开头 |
| 继承父类     | ✅ 必须调用   | super() 之后     |

**代码示例**:

```tsx
// 不继承 — 不需要 super
class Person {
  name: string
  constructor(name: string) {
    this.name = name // ✅ 直接用 this
  }
}

// 继承 — 必须先 super
class GeolocationError extends Error {
  code: number
  constructor(message: string, code: number) {
    super(message) // ✅ 必须先调用
    this.code = code // 然后 this 才可用
  }
}
```

### 问题 #27: 浏览器定位 API 封装详解

**首次记录**: 2026-03-15 03:45
**最近记录**: 2026-03-15 03:45
**记录次数**: 1

**原始问题**:

> 详细解析一下这是如何获取到当前经纬度的，已经各个函数输入的参数是什么，有什么作用？

**相关代码**:

```tsx
const getCurrentPosition = (options = {}) => {
  return new Promise((resolve, reject) => {
    if (!navigator.geolocation) {
      reject(new GeolocationError('浏览器不支持定位功能', 0))
      return
    }
    navigator.geolocation.getCurrentPosition(
      resolve,
      error => {
        const messages = {
          1: '用户拒绝了定位请求',
          2: '无法获取位置信息',
          3: '定位超时',
        }
        reject(new GeolocationError(messages[error.code], error.code))
      },
      options
    )
  })
}
```

**知识点**: JavaScript / 浏览器 API / Promise 封装

**核心解答**:
这是一个**封装函数**，把浏览器原生定位 API（回调风格）包装成 **Promise 风格**。核心是调用 `navigator.geolocation.getCurrentPosition()` 获取设备位置。

**深入理解**:

```tsx
// 核心 API 调用格式
navigator.geolocation.getCurrentPosition(
  参数1: 成功回调,    // 定位成功时调用
  参数2: 失败回调,    // 定位失败时调用
  参数3: 配置选项     // 精度、超时等配置
)
```

**三个参数详解**:

| 参数     | 类型                 | 作用         | 来源                                      |
| -------- | -------------------- | ------------ | ----------------------------------------- |
| 成功回调 | `(position) => void` | 接收定位结果 | 浏览器自动传入 `GeolocationPosition`      |
| 失败回调 | `(error) => void`    | 处理定位错误 | 浏览器自动传入 `GeolocationPositionError` |
| 配置选项 | `object`             | 控制定位行为 | 开发者自定义                              |

**成功回调收到的数据**:

```tsx
position = {
  coords: {
    latitude: 39.9042, // 纬度
    longitude: 116.4074, // 经度
    accuracy: 65, // 精度（米）
    altitude: null, // 海拔
    heading: null, // 方向
    speed: null, // 速度
  },
  timestamp: 1710460800000, // 获取时间
}
```

**三个错误码**:

| 错误码 | 常量名                 | 含义     | 触发场景                |
| ------ | ---------------------- | -------- | ----------------------- |
| 1      | `PERMISSION_DENIED`    | 用户拒绝 | 用户点击"阻止"按钮      |
| 2      | `POSITION_UNAVAILABLE` | 获取失败 | GPS 信号弱、设备不支持  |
| 3      | `TIMEOUT`              | 定位超时 | 超过配置的 timeout 时间 |

**配置选项**:

```tsx
const options = {
  enableHighAccuracy: true, // 高精度模式（耗电更多）
  timeout: 10000, // 超时时间（毫秒）
  maximumAge: 0, // 缓存时间（0 = 强制重新定位）
}
```

**完整执行流程**:

```
调用 getCurrentPosition({ enableHighAccuracy: true })
              ↓
        new Promise 开始
              ↓
    检查浏览器是否支持定位？
        ├─ 不支持 → reject('浏览器不支持')
        └─ 支持 → 继续
              ↓
    调用 navigator.geolocation.getCurrentPosition()
              ↓
        浏览器弹窗请求权限："xxx网站想要获取您的位置"
              ↓
        用户选择？
        ├─ [阻止] → error.code = 1 → reject('用户拒绝')
        └─ [允许] → 浏览器定位（GPS/WiFi/基站）
              ↓
        定位结果？
        ├─ 成功 → resolve(position)
        └─ 失败 → error.code = 2/3 → reject('失败原因')
```

**为什么用 Promise 封装？**

```tsx
// ❌ 原生回调风格 — 难以配合 React Query
navigator.geolocation.getCurrentPosition(
  position => console.log(position.coords.latitude),
  error => console.log(error)
)

// ✅ Promise 风格 — 可用 await，可配合 React Query
const position = await getCurrentPosition()
console.log(position.coords.latitude)

// 配合 React Query
const { data } = useQuery({
  queryKey: ['location'],
  queryFn: async () => {
    const position = await getCurrentPosition()
    return {
      lat: position.coords.latitude,
      lon: position.coords.longitude,
    }
  },
})
```

**代码示例**:

```tsx
// 实际使用
try {
  const position = await getCurrentPosition({ timeout: 10000 })
  console.log('纬度:', position.coords.latitude)
  console.log('经度:', position.coords.longitude)
} catch (error) {
  if (error instanceof GeolocationError) {
    console.log('错误码:', error.code)
    console.log('错误信息:', error.message)
  }
}
```

**学习要点**: ⭐ 核心 API：`getCurrentPosition(成功回调, 失败回调, 配置)`；Promise 封装让回调风格支持 `await`；错误码 1=拒绝，2=失败，3=超时；结果从 `position.coords.latitude/longitude` 获取

### 问题 #28: resolve 的作用 & fetch vs getCurrentPosition

**首次记录**: 2026-03-15 04:00
**最近记录**: 2026-03-15 04:00
**记录次数**: 1

**原始问题**:

> resolve有什么作用?就只是单纯的返回数据？以及fetch(url).then(a => a.json())和getCurrentPosition有什么区别？

**知识点**: JavaScript / Promise / 异步编程

**核心解答**:

- **resolve**：把 Promise 状态变成"成功"，传递结果给 `await`
- **fetch**：从**服务器**获取数据（需要网络）
- **getCurrentPosition**：从**设备**获取位置（GPS，不需要网络）

**深入理解**:

**Promise 三种状态**:

```
        pending（进行中）
           /        \
          /          \
    resolve()      reject()
        ↓              ↓
   fulfilled       rejected
   （已完成）       （已拒绝）

状态只能改变一次，不可逆
```

**resolve 和 await 的关系**:

```tsx
const getLocation = () => {
  return new Promise((resolve, reject) => {
    navigator.geolocation.getCurrentPosition(
      position => resolve(position), // ← 发出结果
      error => reject(error)
    )
  })
}

const position = await getLocation() // ← 接收结果
// resolve(position) 和 await 是"配对"的
```

**类比**:

```
resolve() = 快递员把包裹放进快递柜
await     = 你从快递柜取出包裹
Promise   = 快递柜（连接两端）
```

**fetch vs getCurrentPosition 对比**:

| 对比项   | fetch           | getCurrentPosition |
| -------- | --------------- | ------------------ |
| 数据来源 | 服务器          | 设备（GPS/WiFi）   |
| 需要网络 | ✅ 需要         | ❌ 不需要          |
| 返回类型 | Promise（原生） | Promise（封装后）  |
| 用途     | 获取服务器数据  | 获取用户位置       |

**代码示例**:

```tsx
// fetch — 从服务器获取数据
const weather = await fetch('https://api.weather.com').then(r => r.json())
// weather = { temp: 25, humidity: 60, ... }

// getCurrentPosition — 从设备获取位置
const position = await getCurrentPosition()
// position = { coords: { latitude: 39.9, longitude: 116.4 } }

// 典型场景：先定位，再查天气
const position = await getCurrentPosition()
const weather = await fetch(
  `https://api.weather.com?lat=${position.coords.latitude}&lon=${position.coords.longitude}`
)
```

**为什么都要封装成 Promise？**

```tsx
// ✅ 都是 Promise，可以用同一个流程
const location = await getCurrentPosition()
const weather = await fetch(url).then(r => r.json())

// ❌ 如果 getCurrentPosition 是回调风格，难以配合
navigator.geolocation.getCurrentPosition(
  position => {
    /* 怎么传给 React Query？麻烦！*/
  },
  error => {
    /* 怎么处理错误？麻烦！*/
  }
)
```

**学习要点**: ⭐ resolve 把 Promise 变成成功状态，结果传给 await；fetch 从服务器取数据，getCurrentPosition 从设备取位置；都封装成 Promise 才能配合 React Query

### 问题 #29: await 与 Promise 的结果传递机制

**首次记录**: 2026-03-15 04:15
**最近记录**: 2026-03-15 04:15
**记录次数**: 1

**原始问题**:

> 有关结果传递给 await 解析的详细一点，比如await只是一个会在中间等待promise完成的中间变量是吗？

**知识点**: JavaScript / Promise / async-await

**核心解答**:
**await 不是"变量"，而是一个"暂停标记"**。它会暂停函数执行，等待 Promise 完成，然后"取出" resolve 传递的结果。

**深入理解**:

**await 的两个功能**:

| 功能     | 说明                                   |
| -------- | -------------------------------------- |
| **暂停** | 暂停 async 函数执行，等待 Promise 完成 |
| **取值** | 取出 resolve() 传递的结果              |

```tsx
const position = await getCurrentPosition()
//     ↑            ↑
//   取值          暂停
```

**resolve → await 的传递过程**:

```
Promise 内部                     调用者
    │                              │
    │  resolve(position)           │
    │       │                      │
    │       ▼                      │
    │  ┌─────────────┐             │
    │  │   Promise   │             │
    │  │  (快递柜)    │             │
    │  │  position   │ ◄── 存数据  │
    │  └─────────────┘             │
    │         │                    │
    │         ▼                    │
    │        await                 │
    │         │                    │
    │         ▼                    │
    │   const pos = position       │
    │      (取数据)                 │
```

**详细执行时间线**:

```tsx
async function example() {
  console.log('1. 开始')

  const result = await new Promise(resolve => {
    console.log('2. Promise 内部')
    setTimeout(() => {
      console.log('3. setTimeout 执行')
      resolve('数据') // ← 把 '数据' 放进 Promise
    }, 1000)
  })
  //                    ↑ await 在这里"卡住"等待

  console.log('4. 收到:', result) // result = '数据'
}

example()
console.log('5. 同步代码')

// 输出顺序：
// 1. 开始
// 2. Promise 内部
// 5. 同步代码      ← await 暂停期间，其他代码继续
// （等待 1 秒）
// 3. setTimeout 执行
// 4. 收到: 数据    ← await 取出 resolve 传的值
```

**为什么需要 await？**

```tsx
// ❌ 没有 await
const promise = getCurrentPosition()
console.log(promise) // Promise { <pending> }
// 只拿到 Promise 对象，没有数据！

// ✅ 有 await
const position = await getCurrentPosition()
console.log(position) // { coords: { latitude: 39.9 } }
// 拿到了真正的数据！
```

**Promise 内部如何存储结果（简化版）**:

```tsx
class MyPromise {
  #result = undefined // 隐藏变量，存储结果
  #state = 'pending'

  resolve(value) {
    this.#result = value // 存储结果
    this.#state = 'fulfilled'
  }
}

// await 会：
// 1. 检查 Promise 状态
// 2. 如果 pending，暂停等待
// 3. 如果 fulfilled，取出 #result
// 4. 把 #result 赋值给变量
```

**学习要点**: ⭐ await = 暂停 + 取值；`resolve(数据)` 存进 Promise；`await` 取出数据；传递路径：`resolve` → Promise 内部 → `await` → 变量

### 问题 #30: 什么是自定义 Hook

**首次记录**: 2026-03-15 04:35
**最近记录**: 2026-03-15 04:35
**记录次数**: 1

**原始问题**:

> 为什么可以把这个叫做一个自创的hook？什么是hook？hook一般要做什么事情？只是单纯的传递参数吗？其他API获取的数据也可以写成hook吗？

**知识点**: React / 自定义 Hook / 代码组织

**核心解答**:
**Hook 是 React 提供的一种机制，让你在函数组件中"钩入"状态和生命周期特性。自定义 Hook 本质上就是一个**以 `use` 开头的普通函数**，它内部可以调用其他 Hook。**

**判断自定义 Hook 的两个条件**:

| 条件                   | 说明                                   |
| ---------------------- | -------------------------------------- |
| 1. 函数名以 `use` 开头 | React 约定                             |
| 2. 内部调用了其他 Hook | `useState`、`useEffect`、`useQuery` 等 |

**深入理解**:

**为什么 `useUserLocation` 是自定义 Hook？**

```tsx
export const useUserLocation = () => {  // ✅ 以 use 开头
  return useQuery({...})                // ✅ 内部调用了 useQuery
}
// 两个条件都满足 → 就是自定义 Hook！
```

**Hook 能做什么**:

| 功能             | 示例                                   |
| ---------------- | -------------------------------------- |
| 封装 API 请求    | `useUser()`、`useWeather(lat, lon)`    |
| 封装复杂状态逻辑 | `useForm()`（表单验证）                |
| 封装浏览器 API   | `useLocalStorage()`、`useWindowSize()` |
| 封装订阅/事件    | `useOnlineStatus()`                    |

**不仅仅是传递参数**，Hook 可以封装复杂逻辑：

```tsx
// 示例：表单验证 Hook
export const useForm = <T>(initialValues: T) => {
  const [values, setValues] = useState(initialValues)
  const [errors, setErrors] = useState({})

  const handleChange = (field: keyof T, value: string) => {
    setValues(prev => ({ ...prev, [field]: value }))
    // 自动验证逻辑...
  }

  return { values, errors, handleChange }
}

// 示例：窗口大小 Hook
export const useWindowSize = () => {
  const [size, setSize] = useState({ width: 0, height: 0 })

  useEffect(() => {
    const handleResize = () => setSize({ width: window.innerWidth, height: window.innerHeight })
    window.addEventListener('resize', handleResize)
    return () => window.removeEventListener('resize', handleResize)
  }, [])

  return size
}
```

**其他 API 数据都可以写成 Hook**:

```tsx
// 天气
export const useWeather = (lat: number, lon: number) => {
  return useQuery({ queryKey: ['weather', lat, lon], queryFn: () => fetchWeather(lat, lon) })
}

// 用户信息
export const useUser = (userId: string) => {
  return useQuery({ queryKey: ['user', userId], queryFn: () => fetchUser(userId) })
}

// 搜索建议
export const useSearchSuggestions = (keyword: string) => {
  return useQuery({
    queryKey: ['suggestions', keyword],
    queryFn: () => fetchSuggestions(keyword),
    enabled: keyword.length > 0,
  })
}
```

**自定义 Hook 的好处**:

```tsx
// ❌ 不用 Hook：每个组件重复写
function ComponentA() {
  const { data, isLoading } = useQuery({...})  // 重复
}

// ✅ 用 Hook：简洁复用
function ComponentA() {
  const location = useUserLocation()  // 简洁！
}
```

| 好处 | 说明               |
| ---- | ------------------ |
| 复用 | 相同逻辑多处使用   |
| 简洁 | 组件代码更清晰     |
| 维护 | 修改一处，处处更新 |
| 测试 | Hook 可以单独测试  |

**学习要点**: ⭐ 自定义 Hook = `use` 开头 + 内部调用其他 Hook；用于封装复用逻辑；任何需要复用的逻辑都可以抽成 Hook

### 问题 #31: Zustand vs 自定义 Hook 的职责划分

**首次记录**: 2026-03-15 04:50
**最近记录**: 2026-03-15 04:50
**记录次数**: 1

**原始问题**:

> 如果每个 API 都写成 Hook，那 Zustand 用在哪？它们的作用好像重叠了？

**知识点**: React / 状态管理 / 架构设计

**核心解答**:
**它们解决的是不同类型的状态问题**：

- **React Query（自定义 Hook）**：管理**服务端状态**（API 数据）
- **Zustand**：管理**客户端状态**（用户选择、UI 状态、跨组件共享）

**深入理解**:

**状态类型划分**:

| 状态类型   | 管理方案    | 示例                           |
| ---------- | ----------- | ------------------------------ |
| 服务端状态 | React Query | 天气数据、用户信息、API 数据   |
| 客户端状态 | Zustand     | 用户选择的位置、主题、登录状态 |

**为什么不能混用？**

```tsx
// ❌ 错误：用 Zustand 管理 API 数据
const useWeatherStore = create(set => ({
  weather: null,
  isLoading: false,
  fetchWeather: async (lat, lon) => {
    set({ isLoading: true })
    const data = await fetchWeather(lat, lon) // 手动处理
    set({ weather: data, isLoading: false })
  },
}))
// 问题：没缓存、没刷新、没去重、代码多

// ✅ 正确：用 React Query
const { data: weather, isLoading } = useWeather(lat, lon)
// 自动缓存、刷新、去重，一行搞定
```

**判断标准（三个问题）**:

| 问题                           | 是 → Zustand | 否 → React Query           |
| ------------------------------ | ------------ | -------------------------- |
| 数据来自服务器还是用户操作？   | 用户操作     | 服务器                     |
| 需要跨组件共享吗？             | 是           | React Query 自动共享       |
| 需要持久化到 localStorage 吗？ | 是           | 不需要（服务器数据会过期） |

**完整示例**:

```tsx
// ========== Zustand：客户端状态 ==========
// 用户选择的位置（跨组件共享）
export const useLocationStore = create<LocationStore>(set => ({
  location: { lat: 39.9, lon: 116.4 },
  setLocation: location => set({ location }),
}))

// ========== Hook：服务端状态 ==========
// 天气数据（来自 API）
export const useWeather = (lat: number, lon: number) => {
  return useQuery({
    queryKey: ['weather', lat, lon],
    queryFn: () => fetchWeather(lat, lon),
  })
}

// ========== 组件使用 ==========
function WeatherApp() {
  const { location, setLocation } = useLocationStore() // 客户端状态
  const { data: weather } = useWeather(location.lat, location.lon) // 服务端状态

  return <WeatherCard data={weather} />
}
```

**决策表**:

| 场景               | 解决方案          |
| ------------------ | ----------------- |
| API 数据           | React Query       |
| 用户操作产生的状态 | Zustand           |
| 跨组件共享的状态   | Zustand           |
| 需要持久化的状态   | Zustand + persist |

**学习要点**: ⭐ React Query 管**服务端状态**（API 数据），Zustand 管**客户端状态**（用户选择）；数据来自服务器 → React Query，数据来自用户 → Zustand；不要把 API 数据存到 Zustand

### 问题 #32: React Query enabled 选项的作用

**首次记录**: 2026-03-15 05:15
**最近记录**: 2026-03-15 05:15
**记录次数**: 1

**原始问题**:

> 为什么一旦少了 `enabled: lon !== null && lat !== null`，刷新页面时就不会渲染城市数据到卡片上？

**知识点**: React Query / enabled 选项

**核心解答**:
`enabled` 控制**查询是否启用**。当 `enabled: false` 时查询不执行；当 `enabled` 从 `false` 变成 `true` 时，查询会**自动触发**。

**深入理解**:

**执行时间线对比**:

```
没有 enabled 时：
─────────────────────────────────────
页面刷新 → 组件挂载
lon = null, lat = null（用户位置还在获取中）
    ↓
查询立即执行（没有 enabled 阻止）
queryKey: ['searchCity', null, null]
API: ?location=null,null → 错误！
    ↓
缓存了错误结果 ❌
    ↓
用户位置获取成功，lon = 116, lat = 39
但可能因为错误状态没有正确更新


有 enabled 时：
─────────────────────────────────────
页面刷新 → 组件挂载
lon = null, lat = null
enabled: null !== null → false
    ↓
查询等待，不执行 ✅
    ↓
用户位置获取成功，lon = 116, lat = 39
enabled: 116 !== null → true
    ↓
查询自动触发！✅
API: ?location=116,39 → 成功！
```

**enabled 的行为**:

| enabled 值          | 行为                                  |
| ------------------- | ------------------------------------- |
| `false`             | 查询**不执行**，返回 `status: 'idle'` |
| 从 `false` → `true` | 查询**自动触发**                      |
| `true`              | 正常执行查询                          |

**代码示例**:

```tsx
// 依赖其他数据的查询
const { data: userLocation } = useUserLocation()

const { data: weather } = useWeather(userLocation?.lon, userLocation?.lat)
// ↓ 如果没有 enabled，会用 null 立即查询

// 正确写法：等待条件满足
export const useWeather = (lon: number, lat: number) => {
  return useQuery({
    queryKey: ['weather', lon, lat],
    queryFn: () => fetchWeather(lon, lat),
    enabled: lon != null && lat != null, // ← 等待条件满足
  })
}
```

**类比理解**:

```
enabled 就像一个"开关"

没有 enabled：水龙头一直开着，水还没来就开始放，放出来的是空气（错误）
有 enabled：水龙头先关着，等水来了（条件满足）自动打开，放出来的是水（正确数据）
```

**最佳实践**:

| 场景         | enabled 设置                   |
| ------------ | ------------------------------ |
| 依赖其他数据 | `enabled: !!dependentData`     |
| 条件查询     | `enabled: keyword.length >= 2` |
| 手动触发     | `enabled: false` + `refetch()` |

**学习要点**: ⭐ `enabled: false` 让查询等待；条件满足后自动触发；依赖其他数据时必须用 enabled 等待条件满足，否则会用错误参数立即查询

### 问题 #33: React Query 的响应式机制

**首次记录**: 2026-03-15 05:30
**最近记录**: 2026-03-15 05:30
**记录次数**: 1

**原始问题**:

> enabled 类似 await 吗？数据只传入一次？还是参数变化后重新执行？我只是单纯没更新界面？

**知识点**: React Query / 响应式机制

**核心解答**:
**不是 await，而是"响应式条件"**。数据会**多次传入**（每次渲染），React Query 会**自动检测变化并重新查询**。

**深入理解**:

**组件渲染多次**:

```tsx
function MyComponent() {
  const { data: userLocation } = useUserLocation()
  const { lon, lat } = userLocation ?? { lon: null, lat: null }
  const { data: cityWeather } = useCityWeatherData(lon, lat)

  console.log('渲染', { lon, lat })
  return <WeatherCard data={cityWeather} />
}

// 控制台输出：
// 渲染 { lon: null, lat: null }      ← 第1次渲染
// 渲染 { lon: 116, lat: 39 }         ← 第2次渲染（位置获取成功）
```

**完整时间线**:

```
第1次渲染：lon=null, lat=null
    useQuery 执行
    queryKey: ['searchCity', null, null]
    enabled: false
    → 查询等待，不请求

第2次渲染：lon=116, lat=39  ← 参数变了！
    useQuery 再次执行
    queryKey: ['searchCity', 116, 39]  ← key 变了！
    enabled: true  ← 条件满足了！
    → 自动发起 API 请求
```

**React Query 响应式原理**:

```tsx
useQuery({
  queryKey: ['searchCity', lon, lat], // ← 依赖数组
  enabled: lon !== null,
})

// React Query 会：
// 1. 每次渲染都执行 useQuery
// 2. 检测 queryKey 是否变化
// 3. 检测 enabled 是否变化
// 4. 满足条件时自动触发新查询
```

**关键区别**:

| 概念         | 说明                                   |
| ------------ | -------------------------------------- |
| enabled      | **声明式条件**，不是 await（阻塞等待） |
| 数据传入次数 | **多次**，每次渲染都传入新值           |
| 触发重新查询 | queryKey 变化 + enabled 为 true        |
| 不是界面问题 | 是 **API 请求时机** 的问题             |

**类比**:

```
enabled ≠ await（阻塞等待，代码停在那）
enabled = 条件开关（声明式，React Query 自动检测变化）
```

**学习要点**: ⭐ 数据多次传入（每次渲染）；enabled 是声明式条件非阻塞等待；React Query 自动检测 queryKey 和 enabled 变化；变化时自动触发新查询

### 问题 #34: React Query 的缓存机制

**首次记录**: 2026-03-15 05:45
**最近记录**: 2026-03-15 05:45
**记录次数**: 1

**原始问题**:

> queryKey 中的 lon/lat 是"上一次"传参的内容吗？即使单独开一个 Hook，useQuery 也记录着之前搜索的地点？

**知识点**: React Query / 缓存机制

**核心解答**:
**queryKey 是"当前"的值，不是"上一次"的值**。React Query 会为**每个不同的 queryKey 单独缓存**，多个地点的数据可以同时存在于缓存中。

**深入理解**:

**缓存结构（类似 Map）**:

```
React Query 缓存
─────────────────────────────────────────────
queryKey                     | 缓存数据
─────────────────────────────────────────────
['searchCity', 116, 39]     | { city: '北京' }
['searchCity', 121, 31]     | { city: '上海' }
['searchCity', 113, 23]     | { city: '广州' }
['weather', 116, 39]        | { temp: 25 }
─────────────────────────────────────────────

所有查询结果同时存在！切换回已缓存的 key 直接返回，不重新请求
```

**queryKey 始终是当前值**:

```tsx
function MyComponent() {
  const [lon, setLon] = useState(116)

  const { data } = useQuery({
    queryKey: ['searchCity', lon], // ← 始终是当前渲染时的 lon
  })

  // 第1次渲染：lon=116 → queryKey: ['searchCity', 116]
  // setLon(121) 后
  // 第2次渲染：lon=121 → queryKey: ['searchCity', 121]
  // 不是"上一次"的值，是"当前"的值！
}
```

**缓存生命周期**:

```tsx
const { data } = useQuery({
  queryKey: ['searchCity', lon, lat],
  staleTime: 5 * 60 * 1000, // 5分钟内数据是"新鲜的"
  gcTime: 10 * 60 * 1000, // 10分钟后清除缓存
})

// 时间线：
// 请求 → 缓存数据 → 5分钟内（新鲜期）直接返回缓存
//                → 超过5分钟（过期期）后台刷新
//                → 超过10分钟（清除期）缓存删除
```

**实际场景**:

```tsx
// 用户操作：
// 1. 搜索"北京" → queryKey: ['searchCity', '北京'] → 请求 → 缓存
// 2. 搜索"上海" → queryKey: ['searchCity', '上海'] → 请求 → 缓存
// 3. 再次搜索"北京" → queryKey: ['searchCity', '北京']
//    → 发现已缓存！直接返回，不请求 API ✅
```

**回答总结**:

| 问题                      | 答案                             |
| ------------------------- | -------------------------------- |
| queryKey 是"上一次"的值？ | **不是**，是**当前渲染时**的值   |
| 记住之前搜索的地点？      | **是的**，每个 queryKey 单独缓存 |
| 多个地点同时存在缓存？    | **是的**，缓存是 Map 结构        |

**学习要点**: ⭐ queryKey 是当前值非上次值；每个 key 单独缓存；多个地点可同时存在；重新访问已缓存 key 直接返回不请求

### 问题 #36: React Query 防抖设计

**首次记录**: 2026-03-15 08:00
**最近记录**: 2026-03-15 08:00
**记录次数**: 1

**原始问题**:

> 对于我的 useCityLocation 我应该如何设计防抖？

**知识点**: React / React Query / 防抖（Debounce）

**核心解答**:

**防抖（Debounce）**：用户停止输入一段时间后才执行操作。避免用户每敲一个字就发一次请求，浪费资源。

**实现位置**：在传入 `useQuery` 之前对搜索词进行防抖处理。

**深入理解**:

```tsx
// 当前代码问题：
const data = useQuery({
  queryKey: ['cityLocation', cityName], // cityName 每次变化都触发新请求
  enabled: cityName.length > 1, // 只是控制"何时启用"，不是防抖
})
```

**方案一：useDeferredValue（React 18+ 内置）**

```tsx
import { useDeferredValue } from 'react'

export const useCityLocation = (cityName: string) => {
  const debouncedCityName = useDeferredValue(cityName) // React 自动延迟更新

  const data = useQuery({
    queryKey: ['cityLocation', debouncedCityName],
    queryFn: async () => fetch(url + debouncedCityName),
    enabled: debouncedCityName.length > 1,
  })
  return data
}
```

优点：React 内置，无需额外依赖
缺点：不能自定义延迟时间

**方案二：自定义 useDebouncedValue Hook（推荐）**

```tsx
// hooks/useDebouncedValue.ts
import { useState, useEffect } from 'react'

export const useDebouncedValue = <T>(value: T, delay: number = 500): T => {
  const [debouncedValue, setDebouncedValue] = useState(value)

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebouncedValue(value)
    }, delay)

    return () => clearTimeout(timer) // value 变化时取消之前的定时器
  }, [value, delay])

  return debouncedValue
}
```

```tsx
// useCityLocation.tsx
import { useDebouncedValue } from './useDebouncedValue'

export const useCityLocation = (cityName: string) => {
  const debouncedCityName = useDebouncedValue(cityName, 500) // 500ms 防抖

  const data = useQuery({
    queryKey: ['cityLocation', debouncedCityName],
    queryFn: async () => fetch(url + debouncedCityName),
    enabled: debouncedCityName.length > 1,
  })
  return data
}
```

**效果对比**:

```
用户输入：北 → 北京 → 北京市

【无防抖】输入"北京"→请求，输入"北京市"→请求，共 2 次

【有防抖】输入"北"→等待，输入"北京"→重新等待，输入"北京市"→重新等待，
停止输入 500ms 后→发送请求，共 1 次 ✅
```

**方案选择**:

| 方案               | 延迟时间   | 依赖      | 适用场景     |
| ------------------ | ---------- | --------- | ------------ |
| `useDeferredValue` | React 自动 | React 18+ | 简单场景     |
| 自定义 Hook        | 可自定义   | 无        | 需要精确控制 |
| lodash.debounce    | 可自定义   | lodash    | 复杂场景     |

**学习要点**: ⭐ 防抖 = 用户停止输入一段时间后才执行；用 `setTimeout` + `clearTimeout` 实现；搜索场景推荐 300-500ms；在传入 `useQuery` 前对搜索词防抖

### 问题 #37: useEffect 清理机制与 setTimeout

**首次记录**: 2026-03-15 08:30
**最近记录**: 2026-03-15 08:30
**记录次数**: 1

**原始问题**:

> 为什么 useEffect 要监听 delay？setTimeout 会返回什么？如果 setTimeout 是异步执行的，那不是在执行前就 return 了吗？那 useEffect return 了什么？为什么 return 后面跟着的是 clearTimeout？它有什么作用？

**知识点**: React / useEffect / setTimeout

**核心解答**:

**useEffect 清理函数**：当组件卸载或依赖变化时，React 会执行上一次 effect 返回的清理函数。这是处理副作用的"打扫战场"机制。

**深入理解**:

**问题 1：为什么监听 delay？**

`delay` 决定了定时器的等待时间。如果 `delay` 变化，需要用新延迟重新设置定时器。不监听会导致定时器使用旧值。

**问题 2：setTimeout 返回什么？**

```tsx
const timer = setTimeout(() => {}, 1000)
console.log(timer) // 数字，如 1、2、3...
```

返回 **timer ID（数字）**，用于标识和取消这个定时器。

**问题 3：setTimeout 是异步的，不是执行前就 return 了吗？**

关键理解：`return () => clearTimeout(timer)` 返回的是**函数本身**，不是函数的执行结果！

```tsx
useEffect(() => {
  const timer = setTimeout(...)  // 第1步：创建定时器
  return () => clearTimeout(timer)  // 第2步：返回函数（还没执行！）
  // 第3步：稍后 React 调用这个函数
}, [value])
```

**类比**：

```tsx
// 就像你给朋友留了一把钥匙
useEffect(() => {
  const key = 开门
  return function 锁门() {
    // 这不是现在执行，而是"离开时执行"
    用钥匙锁门
  }
})
```

**问题 4 & 5：clearTimeout 的作用？**

**取消定时器，阻止回调执行**。这是防抖的核心逻辑！

```
【无 clearTimeout】
输入"北"  → timer=1
输入"北京"→ timer=2
输入"北京市"→ timer=3
500ms 后 timer=1,2,3 都触发 → 状态更新 3 次 ❌

【有 clearTimeout】
输入"北"  → timer=1
输入"北京"→ clearTimeout(1) 取消 → timer=2
输入"北京市"→ clearTimeout(2) 取消 → timer=3
500ms 后 timer=3 触发 → 状态更新 1 次 ✅
```

**完整流程**:

```
用户输入 value="北" → 创建 timer=1 → 返回清理函数
用户输入 value="北京" → React 调用清理函数 clearTimeout(1) → 创建 timer=2
用户停止输入 → 500ms 后 timer=2 回调执行 ✅
```

**关键概念表**:

| 概念               | 说明                                |
| ------------------ | ----------------------------------- |
| `setTimeout` 返回  | timer ID（数字），用于取消定时器    |
| `useEffect` return | 返回**清理函数**，不是立即执行      |
| 清理函数执行时机   | 依赖变化或组件卸载时，由 React 调用 |
| `clearTimeout`     | 取消定时器，阻止回调执行            |
| 防抖原理           | 每次输入取消旧定时器+创建新定时器   |

**学习要点**: ⭐ `return () => clearTimeout(timer)` 意思是"下次变化时，先把上次的定时器取消了"；清理函数由 React 在适当时机调用，不是立即执行

### 问题 #38: 函数定义 vs 函数调用 & Timer ID 原理

**首次记录**: 2026-03-15 09:00
**最近记录**: 2026-03-15 09:00
**记录次数**: 1

**原始问题**:

> 为什么 clearTimeout 不会立刻执行？而是在下一次执行？以及计时器的 ID 是如何计算的？是隔一段时间换一个 ID 还是在创建的时候就固定 ID 了？

**知识点**: JavaScript / 函数 / setTimeout

**核心解答**:

**函数定义 ≠ 函数调用**：`return () => clearTimeout(timer)` 返回的是函数本身（定义），不是执行结果。只有加 `()` 才会执行。

**Timer ID 在创建时固定**：`setTimeout` 调用时立即分配 ID，由浏览器内部计数器生成，创建后不会变化。

**深入理解**:

**问题 1：为什么 clearTimeout 不立刻执行？**

```tsx
// ❌ 函数调用 — 立刻执行
return clearTimeout(timer)

// ✅ 函数定义 — 返回函数本身，不执行
return () => clearTimeout(timer)
```

**类比**：

```tsx
// 菜谱 vs 做菜
const 菜谱 = () => 炒番茄炒蛋() // 菜谱（定义）
菜谱() // 做菜（调用）

console.log(菜谱) // 打印函数本身
console.log(菜谱()) // 执行函数，打印结果
```

**执行时间线**：

```
useEffect 执行 → setTimeout 创建 timer=1 → return () => clearTimeout(1)
                                                     ↑
                                              返回函数定义，不执行
React 保存这个函数

value 变化 → React 调用保存的函数 → clearTimeout(1) 执行 ✅
```

**问题 2：Timer ID 如何计算？**

**答案：创建时立即分配，固定不变。**

```tsx
const t1 = setTimeout(() => {}, 1000) // t1 = 1，立即返回
const t2 = setTimeout(() => {}, 1000) // t2 = 2，立即返回
const t3 = setTimeout(() => {}, 1000) // t3 = 3，立即返回

// ID 在创建时分配，不是根据时间
// 即使 t1 回调还没执行，t2 已经是 2 了
```

**浏览器内部机制（简化）**：

```tsx
let timerCounter = 0

function setTimeout(callback, delay) {
  timerCounter++ // 计数器 +1
  const id = timerCounter // 分配 ID
  timerMap.set(id, { callback, executeTime: Date.now() + delay })
  return id // 立即返回
}

function clearTimeout(id) {
  timerMap.delete(id) // 从映射表删除
}
```

**关键区别**：

| 概念         | 说明                     | 示例                           |
| ------------ | ------------------------ | ------------------------------ |
| 函数定义     | 返回函数本身             | `return () => fn()`            |
| 函数调用     | 立刻执行                 | `return fn()`                  |
| Timer ID     | 创建时立即分配，固定不变 | `setTimeout()` 返回 1, 2, 3... |
| clearTimeout | 从映射表删除，ID 不变    | `clearTimeout(1)` 后 ID 还是 1 |

**学习要点**: ⭐ `() => fn()` 是定义（不执行），`fn()` 是调用（立即执行）；Timer ID 创建时立即分配，固定不变，不会复用

### 问题 #39: useEffect 清理函数的执行时机

**首次记录**: 2026-03-15 09:30
**最近记录**: 2026-03-15 09:30
**记录次数**: 1

**原始问题**:

> 为什么监听的属性变化了就会执行 return 返回的函数，这有些反直觉不是吗？

**知识点**: React / useEffect / 生命周期

**核心解答**:

**useEffect 的设计哲学**：每次新 effect 执行前，先清理上一次的 effect。就像"离开房间前要打扫，然后才能进入新房间"。

**深入理解**:

**useEffect 生命周期**：

```
1. 组件挂载 → 执行 effect
2. 依赖变化 → 先执行清理函数 → 再执行新的 effect
3. 组件卸载 → 执行清理函数
```

**为什么这样设计？**

清理函数的作用是"打扫战场"——取消旧的定时器、移除事件监听器、取消网络请求。如果在开始新 effect 前不清理旧的，就会出问题：

```tsx
// 无清理机制
useEffect(() => {
  setTimeout(() => fetchData(value), 500)
}, [value])
// value 从 A→B → 发送两个请求，A 的结果可能覆盖 B ❌

// 有清理机制
useEffect(() => {
  const timer = setTimeout(() => fetchData(value), 500)
  return () => clearTimeout(timer) // 取消旧的
}, [value])
// value 从 A→B → 取消 A 的请求，只发送 B ✅
```

**类比：住酒店**

```
入住 → 拿钥匙（effect 执行）
换房 → 先退旧钥匙（清理函数）→ 再拿新钥匙（新 effect）
退房 → 退还钥匙（卸载时清理）
```

**React 内部实现（简化）**：

```tsx
let previousCleanup = null

function runEffect(effect) {
  if (previousCleanup) {
    previousCleanup() // 先清理旧的 ← 这就是为什么清理函数会执行！
  }
  previousCleanup = effect() // 执行新 effect，保存清理函数
}
```

**执行时间线**：

```
value=A → effect 执行，打印 "effect: A"，保存 cleanup_A
value=B → cleanup_A() 执行，打印 "cleanup: A"
          → effect 执行，打印 "effect: B"，保存 cleanup_B
value=C → cleanup_B() 执行，打印 "cleanup: B"
          → effect 执行，打印 "effect: C"，保存 cleanup_C
卸载    → cleanup_C() 执行，打印 "cleanup: C"
```

**为什么"反直觉"？**

习惯的直觉：`return 函数` → 函数执行完就结束了
React 的设计：`return 函数` → React 保存它 → 适当时机调用

这是 React 的"声明式"思想：你告诉 React "如果要清理，请执行这个函数"，React 决定何时调用。

**学习要点**: ⭐ useEffect 清理时机：每次新 effect 执行前先清理旧的；设计目的：防止旧副作用干扰新的；类比：换房间前先退旧钥匙；声明式：你提供清理函数，React 决定何时调用

**精确理解**:

```tsx
useEffect(() => {
  // effect 开始
  return () => {
    // effect 结束时调用
  }
  //      ↑ 告诉 React："这个 effect 结束时，请执行这个清理函数"
}, [value])
```

| 场景     | "结束"的含义                     |
| -------- | -------------------------------- |
| 依赖变化 | 新 effect 开始前，旧 effect 结束 |
| 组件卸载 | 组件从 DOM 移除前                |

**一句话总结**：`return 清理函数` 就是告诉 React "这个 effect 结束时请帮我打扫战场"

### 问题 #47: Zustand 中 get 的作用

**首次记录**: 2026-03-17 09:30
**最近记录**: 2026-03-17 09:30
**记录次数**: 1

**原始问题**:

> zustand中 get有什么作用?

**知识点**: Zustand / 状态管理

**核心解答**:

`get` 是 Zustand 提供的**读取当前状态**的方法。`set` 用于更新状态，`get` 用于获取当前状态值。

**深入理解**:

在 Zustand 的 `create` 函数中，回调函数接收两个参数：

```tsx
create<State>((set, get) => ({
  // set - 更新状态
  // get - 读取当前状态
}))
```

**为什么需要 `get`？**

因为 `set` 的更新函数中，你无法直接访问 `this.state`——Zustand 不是 React 的 class 组件。如果你想基于**当前状态**计算新状态，就必须用 `get()`。

**关键区别**:

| 方法  | 作用         | 类比                      |
| ----- | ------------ | ------------------------- |
| `set` | 更新状态     | 相当于 `setState`         |
| `get` | 读取当前状态 | 相当于 `useStore().state` |

**代码示例**:

结合你的项目 `useFavoritesCity.tsx`，假设要实现"添加城市（避免重复）"：

```tsx
export const useFavoritesCity = create<FavoritesCityList>((set, get) => ({
  list: [],

  addCity: (city: City) => {
    // ✅ 用 get() 读取当前状态
    const currentList = get().list
    const exists = currentList.find(c => c.id === city.id)

    if (exists) return // 已存在，不重复添加

    set({ list: [...currentList, city] })
  },

  removeCity: (id: string) => {
    const currentList = get().list
    set({ list: currentList.filter(c => c.id !== id) })
  },
}))
```

**另一种写法（使用 `set` 回调）**:

`set` 也支持回调形式，可以避免显式调用 `get`：

```tsx
// 写法1：使用 get()
addCity: city => {
  const currentList = get().list
  set({ list: [...currentList, city] })
}

// 写法2：使用 set 回调（更简洁）
addCity: city => {
  set(state => ({ list: [...state.list, city] }))
}
```

**什么时候必须用 `get`？**

当你需要先读取状态做一些判断，再决定是否更新：

```tsx
addCity: city => {
  const currentList = get().list

  // 复杂逻辑：判断是否存在、限制数量等
  if (currentList.length >= 10) {
    alert('最多收藏10个城市')
    return
  }
  if (currentList.find(c => c.id === city.id)) {
    return // 已存在
  }

  set({ list: [...currentList, city] })
}
```

**学习要点**: ⭐ `set` = 写入状态，`get` = 读取当前状态；在 action 方法中需要基于当前状态计算时，用 `get()` 获取；简单更新用 `set(state => ...)` 回调即可

### 问题 #20: 为什么更新对象要用展开运算符

**首次记录**: 2026-03-17 10:30
**最近记录**: 2026-03-17 10:30
**记录次数**: 1

**原始问题**:

> `{ ...c, hasFavorite: !c.hasFavorite }` 我直接 `c.hasFavorite = !c.hasFavorite` 不行吗？

**知识点**: React / 不可变性 / 对象更新

**核心解答**:

**React 要求状态必须是"不可变"的**——你不能直接修改原对象，而是要创建一个新对象。直接修改会导致 React 无法检测到变化，界面不会更新。

**深入理解**:

React 比较的是"对象引用"，不是"对象内容"：

```tsx
// ❌ 直接修改 — 同一个对象引用
const oldObj = { id: 1, hasFavorite: false }
oldObj.hasFavorite = true
oldObj === oldObj // true！React 认为"没变化"

// ✅ 创建新对象 — 不同的引用
const oldObj = { id: 1, hasFavorite: false }
const newObj = { ...oldObj, hasFavorite: true }
oldObj === newObj // false！React 检测到变化
```

**两种写法对比**:

```tsx
// ❌ 错误：直接修改
list.map(c => {
  c.hasFavorite = !c.hasFavorite // 直接改原对象
  return c // 返回同一个引用
})

// ✅ 正确：创建新对象
list.map(
  c =>
    c.id === id
      ? { ...c, hasFavorite: !c.hasFavorite } // 新对象，新引用
      : c // 不变的项返回原对象即可
)
```

**展开运算符的作用**:

```tsx
{ ...c, hasFavorite: !c.hasFavorite }

// 相当于：
{
  id: c.id,
  lon: c.lon,
  lat: c.lat,
  hasFavorite: !c.hasFavorite,  // 只有这个被覆盖
}
```

**直接修改的后果**:

| 后果       | 说明                |
| ---------- | ------------------- |
| 不重新渲染 | React 检测不到变化  |
| 调试困难   | 状态变化无法追踪    |
| 优化失效   | React.memo 等失效   |
| 奇怪 bug   | 违反 React 设计原则 |

**学习要点**: ⭐ React 状态不可变 = 永远创建新对象；展开运算符 `{ ...old, key: newValue }` 是标准写法；直接修改导致 React 检测不到变化

### 问题 #21: Zustand 中 set(state => ...) 与 get() 的区别

**首次记录**: 2026-03-17 11:00
**最近记录**: 2026-03-17 11:00

> set 中的 state 返回的是什么？这和 get 有什么区别？

**知识点**: Zustand / 状态管理

**核心解答**:

**它们获取的是同一个东西——当前状态**。区别只是"获取方式"不同：`get()` 是显式调用，`set(state => ...)` 是通过参数隐式传入。

**深入理解**:

```tsx
// 这两种写法获取的状态完全一样

// 方式1：用 get()
addCity: (city: City) => {
  const currentList = get().list // 显式调用
  set({ list: [...currentList, city] })
}

// 方式2：用 set(state => ...)
addCity: (city: City) => {
  set(state => ({ list: [...state.list, city] }))
  //            ↑ state 就是当前状态，和 get() 返回的一样
}
```

**验证它们是同一个对象**:

```tsx
someAction: () => {
  console.log(get())           // { list: [...], ... }
  set(state => {
    console.log(state === get()) // true！完全相同
    return { ... }
  })
}
```

| 场景             | 推荐写法                   |
| ---------------- | -------------------------- |
| 需要先判断再更新 | `get()` 更清晰             |

```tsx
// 需要判断 → 用 get()
addCity: (city: City) => {
  const currentList = get().list
  if (currentList.find(c => c.id === city.id)) return // 判断
  set({ list: [...currentList, city] })
}

// 直接更新 → 用 set(state => ...)
removeCity: (id: string) => {
  set(state => ({ list: state.list.filter(c => c.id !== id) }))
}
```

**学习要点**: ⭐ `state` 和 `get()` 返回同一个状态对象；需要判断用 `get()`，直接更新用 `set(state => ...)`


```tsx
// 方法1：用 .length（推荐）
if (citys.length === 0) {
```

SVG 图标通过 `fill`（填充）实现实心，通过 `stroke`（描边）+ `fill="none"` 实现空心。可以通过 props 控制两种状态。

**深入理解**:

**fill vs stroke 对比**：

| 属性          | 作用     | 实心图标       | 空心图标       |
| ------------- | -------- | -------------- | -------------- |
| `fill`        | 填充颜色 | `currentColor` | `none`         |
| `stroke`      | 描边颜色 | 不需要         | `currentColor` |
| `strokeWidth` | 描边宽度 | 0              | 1.5~2          |

**代码示例**：

```tsx
interface FavoriteIconProps {
  className?: string
  size?: number
  filled?: boolean  // 控制是否实心
}

export default function FavoriteIcon({
  className,
  size = 28,
  filled = true
}: FavoriteIconProps) {
  return (
    <svg
      width={size}
      height={size}
      viewBox="0 0 28 28"
      fill="none"
      xmlns="http://www.w3.org/2000/svg"
      className={className}
    >
      <path
        fillRule="evenodd"
        clipRule="evenodd"
        d="M..."
        fill={filled ? "currentColor" : "none"}
        stroke="currentColor"
        strokeWidth={filled ? 0 : 1.5}
      />
    </svg>
  )
}

// 使用
<FavoriteIcon filled={true} className="text-yellow-400" />   // 实心
<FavoriteIcon filled={false} className="text-gray-400" />    // 空心
```

**学习要点**: ⭐ 实心用 `fill`，空心用 `stroke` + `fill="none"`。通过 props 控制状态实现组件复用

### 问题 #56: Zustand Store 命名规范

**首次记录**: 2026-03-17 17:15
**最近记录**: 2026-03-17 17:15
**记录次数**: 1

**原始问题**:

> 我的这个 zustand 应该如何命名才符合规范？

**相关代码**:

```tsx
export const useNowCityId = (code: string) => {
  // ❌ 这不是 Hook，是创建 store 的函数
}
```

**知识点**: Zustand / 命名规范 / React Hooks

**核心解答**:

Zustand store 是状态管理 Hook，命名应遵循：以 `use` 开头（表明是 Hook），以 `Store` 结尾（表明是 Zustand store）。

**深入理解**:

**命名对照表**：

| 类型           | 命名格式                  | 示例                |
| -------------- | ------------------------- | ------------------- |
| Store Hook     | `use` + 名词 + `Store`    | `useCityIdStore`    |
| Store 工厂函数 | `create` + 名词 + `Store` | `createCityIdStore` |
| Selector Hook  | `use` + 功能描述          | `useCurrentCity`    |

**代码示例**：

```tsx
// ✅ 模式1：直接导出 Store Hook（推荐）
export const useCityIdStore = create<CityIdState>(set => ({
  cityId: '',
  setCityId: (id: string) => set({ cityId: id }),
}))

// 使用
const { cityId, setCityId } = useCityIdStore()

// ✅ 模式2：工厂函数创建多个 store
export const createCityIdStore = (initialId: string) =>
  create<CityIdState>(set => ({
    cityId: initialId,
    setCityId: (id: string) => set({ cityId: id }),
  }))

// ✅ 模式3：派生 Hook
export const useCurrentCityName = () => {
  return useCityIdStore(state => state.cityId.toUpperCase())
}
```

**关键区别**：

| 概念         | 返回值        | 命名             |
| ------------ | ------------- | ---------------- |
| **Store**    | Zustand Hook  | `useXxxStore`    |
| **Hook**     | 普通值/函数   | `useXxx`         |
| **工厂函数** | Store Creator | `createXxxStore` |

**学习要点**: ⭐ Store 以 `Store` 结尾，Hook 以 `use` 开头。`useXxxStore` = 这是一个 Zustand Store

### 问题 #58: Zustand 选择器订阅机制

**首次记录**: 2026-03-17 18:00
**最近记录**: 2026-03-17 18:00
**记录次数**: 1

**原始问题**:

> 为什么我不订阅 list 就不会更新按钮状态？

**相关代码**:

```tsx
const isInList = useFavoritesCityStore(state => state.isInList)
// 不订阅 list
const isFavorited = isInList(cityId) // ❌ 永远是初始值！
```

**知识点**: Zustand / React / 状态管理

**核心解答**:

Zustand 只订阅你选择的状态。`isInList` 是函数引用（永远不变），`isInList(id)` 的返回值依赖 `list`。不订阅 `list` 就不会触发重渲染。

**深入理解**:

**执行流程分析**：

```
订阅 isInList（函数引用）
    ↓
list 变化
    ↓
isInList 函数引用没变（还是同一个函数）
    ↓
Zustand 认为没有变化，不触发重渲染
    ↓
isInList(id) 不会重新计算
    ↓
按钮状态不更新！
```

**选择器对比**：

| 写法                            | 订阅什么  | 重渲染时机             |
| ------------------------------- | --------- | ---------------------- |
| `state => state.isInList`       | 函数引用  | 永远不会               |
| `state => state.list`           | list 数组 | list 任何变化时        |
| `state => state.list.some(...)` | 布尔结果  | 匹配结果变化时（最优） |

**正确做法**：

```tsx
// ✅ 方案1：订阅 list
const list = useFavoritesCityStore(state => state.list)
const isFavorited = list.some(c => c.id === cityId)

// ✅ 方案2：派生选择器（推荐）
const isFavorited = useFavoritesCityStore(state => state.list.some(c => c.id === cityId))
```

**学习要点**: ⭐ Zustand 只订阅你选择的。函数引用不变 ≠ 函数返回值不变。要订阅"结果"而不是"函数"

### 问题 #59: onBlur 在非可聚焦元素上不生效

**首次记录**: 2026-03-17 18:15
**最近记录**: 2026-03-17 18:15
**记录次数**: 1

**原始问题**:

> 为什么我点击其他地方后我的 onBlur 不会执行？列表不会被隐藏？

**相关代码**:

```tsx
<li onBlur={() => setIsHide(false)}>
  {' '}
  // ❌ li 不可聚焦，onBlur 永远不触发
  <span>
    {city.adm2}-{city.name}
  </span>
</li>
```

**知识点**: React / DOM事件 / 可访问性

**核心解答**:

`onBlur` 只有在元素**失去焦点**时才触发。`<li>` 默认**不是可聚焦元素**，它永远不会获得焦点，所以 `onBlur` 永远不会触发。

**深入理解**:

**可聚焦元素**（默认支持 focus）：

- `<button>`, `<input>`, `<a>`, `<textarea>`, `<select>`

**不可聚焦元素**（默认不支持 focus）：

- `<div>`, `<li>`, `<span>`, `<p>`

**解决方案对比**：

| 方案               | 代码                                          | 优缺点              |
| ------------------ | --------------------------------------------- | ------------------- |
| tabIndex + onBlur  | `<li tabIndex={0} onBlur={...}>`              | 简单，但体验一般    |
| useEffect 点击外部 | `document.addEventListener('mousedown', ...)` | 标准做法，体验好 ✅ |

**正确做法（useEffect 点击外部）**：

```tsx
const listRef = useRef<HTMLDivElement>(null)

useEffect(() => {
  const handleClickOutside = (e: MouseEvent) => {
    if (listRef.current && !listRef.current.contains(e.target as Node)) {
      setIsHide(true)
    }
  }

  if (!isHide) {
    document.addEventListener('mousedown', handleClickOutside)
  }

  return () => {
    document.removeEventListener('mousedown', handleClickOutside)
  }
}, [isHide])

return <div ref={listRef}>{/* 列表内容 */}</div>
```

**学习要点**: ⭐ `onBlur` 只对可聚焦元素生效。`<div>`、`<li>` 默认不可聚焦。点击外部关闭用 `useEffect` + `mousedown` 监听

### 问题 #61: Zustand persist 持久化中间件

**首次记录**: 2026-03-17 18:45

> persist 是什么？是 Zustand 特有的吗？和 Zustand 有什么配合？

**知识点**: Zustand / 状态持久化 / 中间件

**核心解答**:

`persist` 是 Zustand 官方提供的中间件，用于将 store 状态自动保存到 localStorage，实现数据持久化。页面刷新后数据不丢失。

**深入理解**:

**数据流**：

```
用户操作 → set() → Store 更新 → persist 自动触发 → localStorage 保存
                                                    ↓
页面刷新 → persist 读取 localStorage → JSON.parse → 恢复 Store 状态
```

**基本用法**：

```typescript
import { create } from 'zustand'
import { persist } from 'zustand/middleware'

export const useStore = create<Store>()(
  persist(
    set => ({
      count: 0,
      increment: () => set(s => ({ count: s.count + 1 })),
    }),
    {
      name: 'my-storage', // localStorage key
    }
  )
)
```

**常用配置**：

```typescript
persist(
  (set) => ({ ... }),
  {
    name: 'storage-key',                              // 必填
    storage: createJSONStorage(() => sessionStorage), // 换存储方式
    partialize: (state) => ({ count: state.count }),  // 只存部分字段
    version: 1,                                       // 版本号
  }
)
```

**与普通 Zustand 对比**：

| 对比项   | 普通 Zustand | Zustand + persist |
| -------- | ------------ | ----------------- |
| 刷新页面 | 数据丢失 ❌  | 数据保留 ✅       |
| 存储位置 | 内存         | localStorage      |
| 代码量   | 少           | 多一层包裹        |

**学习要点**: ⭐ persist 是 Zustand 中间件，自动保存状态到 localStorage。用法：`create()(persist(store, { name: 'key' }))`

使用 Zustand 管理主题状态 + Tailwind `dark:` 前缀 + localStorage 持久化，实现全站主题切换功能。

**深入理解**:

**实现架构**：

```
用户点击切换 → Zustand Store 更新 → <html> 添加 dark 类 → Tailwind dark: 生效 → localStorage 持久化
```

**核心要点**：

| 要点       | 说明                                        |
| ---------- | ------------------------------------------- |
| 状态管理   | Zustand + persist 中间件                    |

| 初始化同步 | useEffect 同步 store 到 DOM                 |

**深入理解**:

| 对比项       | Tailwind v3（旧版）                   | Tailwind v4（你的版本） |
| ------------ | ------------------------------------- | ----------------------- |
| 配置文件     | 需要 `tailwind.config.js`             | ❌ **不需要**           |
| CSS 写法     | `@tailwind base/components/utilities` | `@import 'tailwindcss'` |
| 暗色模式配置 | 需设置 `darkMode: 'class'`            | **默认启用**            |
| 自定义主题   | JS 配置文件                           | CSS 中用 `@theme` 指令  |
| 安装方式     | `tailwindcss postcss autoprefixer`    | `@tailwindcss/vite`     |

**你的项目配置**：

```json
// package.json
"tailwindcss": "^4.2.1",
"@tailwindcss/vite": "^4.2.1"
```

```tsx
// vite.config.ts
import tailwindcss from '@tailwindcss/vite'
plugins: [react(), tailwindcss()]
```

```css
/* src/index.css */
@import 'tailwindcss';
```

**代码示例**：

```tsx
// ✅ v4 中直接使用 dark: 前缀，无需任何配置
;<div className="bg-white dark:bg-gray-900 text-black dark:text-white">
  <h1>天气咨询</h1>
</div>

// 触发暗色模式
const [isDark, setIsDark] = useState(false)

useEffect(() => {

}, [isDark])
```

```css
/* src/index.css */

  --font-sans: 'Inter', sans-serif;
}
```

**学习要点**: ⭐ Tailwind v4 **不需要** `tailwind.config.js`；暗色模式**默认启用**；自定义配置在 CSS 中用 `@theme` 指令；看教程先确认 Tailwind 版本，v3 和 v4 差异很大

### 问题 #67: document.documentElement 指向哪个元素

**首次记录**: 2026-03-18 21:20
**最近记录**: 2026-03-18 21:20
**记录次数**: 1

**原始问题**:

> `document.documentElement.classList.add('dark')` 是给哪个标签添加了 dark 类？是 App 还是 App 中的子标签？

**知识点**: DOM / JavaScript / 主题切换

**核心解答**:

`document.documentElement` 返回文档的**根元素**，即 **`<html>` 标签**。不是 `<App>` 组件，也不是子标签。

**深入理解**:

**HTML 文档结构**：

```html
<!DOCTYPE html>
<html lang="zh-CN">
  <!-- ← document.documentElement 指向这里 -->
  <head>
    <meta charset="UTF-8" />
    <title>天气咨询</title>
  </head>
  <body>
    <div id="root">
      <!-- React 应用挂载点 -->
      <App />
      <!-- 你的组件渲染在这里 -->
    </div>
  </body>
</html>
```

**执行后的效果**：

```html
<!-- theme === 'dark' 时 -->
<html lang="zh-CN" class="dark">
  <!-- Tailwind 检测到 dark 类，dark: 前缀生效 -->
</html>
```

**选择器对照表**：

| 选择器                            | 指向的元素        | 用途                     |
| --------------------------------- | ----------------- | ------------------------ |
| `document.documentElement`        | `<html>`          | 全局样式控制（主题切换） |
| `document.body`                   | `<body>`          | 页面级样式控制           |
| `document.getElementById('root')` | `<div id="root">` | React 挂载点             |

**代码示例**：

```tsx
// ✅ 正确：监听 theme 变化
useEffect(() => {
  if (theme === 'dark') {
    document.documentElement.classList.add('dark')
  } else {
    document.documentElement.classList.remove('dark')
  }
}, [theme]) // 必须添加 theme 依赖！
```

**为什么在 `<html>` 上添加 dark 类？**

Tailwind 默认检查 `<html>` 元素上的 `dark` 类。当检测到 `class="dark"` 时，所有 `dark:` 前缀的样式才会生效。

**学习要点**: ⭐ `document.documentElement` = `<html>` 元素；主题切换要在 `<html>` 上加 `dark` 类；`useEffect` 要监听 `theme` 才能响应切换

### 问题 #68: document.documentElement 在哪里都能调用吗

**首次记录**: 2026-03-18 21:25
**最近记录**: 2026-03-18 21:25
**记录次数**: 1

**原始问题**:

> 也就是说这个函数我不管在哪里调用都可以给 html 添加 dark 类喽？

**知识点**: DOM / JavaScript / React useEffect

**核心解答**:

**是的，基本上在任何地方调用都可以**。`document` 是浏览器的全局对象，只要在浏览器环境中，任何 JavaScript 代码都能访问并修改 `document.documentElement`。

**深入理解**:

**可以在哪里调用**：

| 位置             | 能否调用  | 说明                                |
| ---------------- | --------- | ----------------------------------- |
| `useEffect` 中   | ✅ 推荐   | React 标准做法，组件挂载后执行      |
| 事件处理函数中   | ✅ 可以   | 点击切换主题时调用                  |
| Zustand store 中 | ✅ 可以   | 状态更新时同步 DOM                  |
| 组件函数体中     | ⚠️ 不推荐 | SSR 时会报错（服务端没有 document） |
| `index.tsx` 顶层 | ⚠️ 不推荐 | 可能比 React 渲染更早执行           |

**代码示例**：

```tsx
// ✅ 方式1：在 Zustand store 中同步（推荐）
export const useThemeStore = create<ThemeStore>()(
  persist(
    (set, get) => ({
      theme: 'light',
      toggleTheme: () => {
        const newTheme = get().theme === 'light' ? 'dark' : 'light'
        set({ theme: newTheme })
        // 直接在这里同步 DOM
        document.documentElement.classList.toggle('dark', newTheme === 'dark')
      },
    }),
    { name: 'theme-storage' }
  )
)

// ✅ 方式2：在 useEffect 中同步
useEffect(() => {
  document.documentElement.classList.toggle('dark', theme === 'dark')
}, [theme])

// ❌ 不推荐：直接在组件函数体中调用
function App() {
  document.documentElement.classList.add('dark') // SSR 会报错！
  return <div>...</div>
}
```

**为什么通常放在 useEffect 中**：

`useEffect` 保证只在**浏览器环境**执行，不会在服务端渲染（SSR）时报错。服务端没有 `document` 对象，直接访问会抛出 `ReferenceError`。

**学习要点**: ⭐ `document.documentElement` 全局可访问；推荐在 `useEffect` 或事件处理函数中调用；放在 store 的 `toggleTheme` 中同步是常见模式

### 问题 #69: Zustand 主题切换的最佳实践

**首次记录**: 2026-03-18 21:30
**最近记录**: 2026-03-18 21:30
**记录次数**: 1

**原始问题**:

> 解析一下 `updateDocumentTheme(theme)`，以及有没有更加优雅的解决方法？

**相关代码**:

```tsx
export const useThemeStore = create<ThemeStore>()(
  persist(
    (set, get) => ({
      theme: 'light',
      toggleTheme: () => {
        const newTheme = get().theme === 'light' ? 'dark' : 'light'
        set({ theme: newTheme })
        updateDocumentTheme(newTheme) // 在 store 中操作 DOM
      },
    }),
    { name: 'theme-storage' }
  )
)

function updateDocumentTheme(theme: Theme) {
  if (theme === 'dark') {
    document.documentElement.classList.add('dark')
  } else {
    document.documentElement.classList.remove('dark')
  }
}
```

**知识点**: React / Zustand / 架构设计

**核心解答**:

`updateDocumentTheme` 是 DOM 同步辅助函数，给 `<html>` 添加/移除 `dark` 类。当前方案在 store 中直接操作 DOM，混合了状态管理和副作用，不够优雅。

**深入理解**:

**当前方案的问题**：

| 问题     | 说明                                         |
| -------- | -------------------------------------------- |
| 职责混乱 | Store 既管理状态又操作 DOM，违反单一职责     |
| 重复逻辑 | `useEffect` 和 `toggleTheme` 都在做 DOM 同步 |
| 维护困难 | 改主题逻辑需要改两个地方                     |

**更优雅的方案（useEffect 统一处理）**：

```tsx
// Store 只管状态，不管 DOM
export const useThemeStore = create<ThemeStore>()(
  persist(
    set => ({
      theme: 'light',
      toggleTheme: () => {
        set(state => ({
          theme: state.theme === 'light' ? 'dark' : 'light',
        }))
      },
      setTheme: theme => set({ theme }),
    }),
    { name: 'theme-storage' }
  )
)

// App.tsx 中统一处理 DOM 同步
function App() {
  const theme = useThemeStore(state => state.theme)

  // 监听 theme 变化，自动同步到 DOM
  useEffect(() => {
    document.documentElement.classList.toggle('dark', theme === 'dark')
  }, [theme]) // 必须有 theme 依赖！

  return <div>...</div>
}
```

**方案对比**：

| 方案      | DOM 同步位置 | 职责分离 | 推荐度     |
| --------- | ------------ | -------- | ---------- |
| 当前方案  | store 内部   | ❌ 混合  | ⭐⭐       |
| useEffect | React 组件   | ✅ 分离  | ⭐⭐⭐⭐⭐ |
| subscribe | store 外部   | ✅ 分离  | ⭐⭐⭐⭐   |

**简洁写法**：

```tsx
// 这两种写法等价
document.documentElement.classList.toggle('dark', theme === 'dark')

// 等价于
if (theme === 'dark') {
  document.documentElement.classList.add('dark')
} else {
  document.documentElement.classList.remove('dark')
}
```

**学习要点**: ⭐ 职责分离：Store 管状态，useEffect 管 DOM；`useEffect(() => {...}, [theme])` 监听变化自动同步；`classList.toggle('dark', condition)` 更简洁
